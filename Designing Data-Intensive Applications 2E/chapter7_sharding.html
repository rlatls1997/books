Chapter 7 · Designing Data-Intensive Applications

샤딩
Sharding

데이터나 쓰기 처리량이 한 노드가 감당하기엔 너무 클 때, 데이터를 더 작은 샤드(파티션)로 쪼개 서로 다른 노드에 둔다. 각 레코드는 정확히 하나의 샤드에 속한다.

키 범위
정렬된 키 범위를 샤드에 할당 — 범위 질의에 유리
키 해시
키를 해싱해 분산 — 부하 균등, 범위 질의 비효율
+ 복제
각 샤드는 여러 노드에 복제돼 내결함성 확보
데이터 분산의 두 방식

복제 + 샤딩

  • 복제(6장) — 같은 데이터의 복사본을 여러 노드에.
  • 샤딩(이 장) — 데이터를 쪼개 다른 샤드를 다른 노드에.

보통 둘을 함께 쓴다. 단일 리더 복제와 결합하면 — 각 샤드의 리더가 한 노드에, 팔로워는 다른 노드에. 한 노드가 어떤 샤드의 리더이자 다른 샤드의 팔로워일 수 있다.

용어 정리

"샤드"는 소프트웨어마다 이름이 다르다

Kafka · partitionCockroachDB · rangeHBase·TiDB · regionCouchbase · vBucketRiak · vnodeCassandra · token-rangeBigtable·YugabyteDB·ScyllaDB · tablet
일부 DB는 파티션과 샤드를 구분한다. PostgreSQL에서 파티셔닝은 큰 테이블을 같은 머신의 여러 파일로 쪼개는 것(파티션 통째 삭제가 빠름), 샤딩은 여러 머신에 분산. 많은 시스템에선 둘이 동의어다.
어원: 온라인 RPG Ultima Online에서 마법 수정이 조각(shard)나며 각 조각이 게임 세계의 복사본을 비췄다는 설. 한편 네트워크 파티션(netsplit)과는 전혀 무관하다.
샤딩의 장단점

확장성을 얻고 복잡성을 치른다

왜 하는가 — 확장성

데이터·쓰기 처리량이 한 노드를 넘으면 여러 노드로 분산. 수평 확장(scale-out)의 핵심 도구. 워크로드를 균등 분할하면 노드 추가로 용량을 키운다. (읽기만 문제면 읽기 확장으로 충분)

왜 신중한가 — 복잡성

파티션 키를 정해야 하고(같은 키는 같은 샤드), 키를 모르면 모든 샤드를 비효율적으로 뒤져야 한다. 샤딩 스킴은 바꾸기 어렵다. 보조 인덱스·조인은 까다롭고, 여러 샤드 쓰기는 분산 트랜잭션(보통 훨씬 느림)이 필요하다.

한 머신이 감당할 수 있다면(요즘 머신은 많은 걸 할 수 있다!) 샤딩을 피하고 단일 샤드가 낫다.
단일 머신 내에서도 샤딩을 쓴다 — CPU 코어당 단일 스레드 프로세스 하나로 병렬성·NUMA를 활용. Redis · VoltDB · FoundationDB
멀티테넌시를 위한 샤딩

테넌트마다(또는 묶어서) 샤드

SaaS·클라우드는 흔히 멀티테넌트 — 각 테넌트(고객)는 독립된 데이터셋을 가진다. 테넌트별 샤딩의 이점:

자원 격리

한 테넌트의 비싼 연산이 다른 테넌트 성능에 영향을 덜 준다.

권한 격리

접근 제어 버그가 있어도 다른 테넌트 데이터에 실수로 접근할 위험↓.

셀 기반 아키텍처

서비스+스토리지를 자족적 셀로 묶어 장애 격리 — 한 셀 장애가 다른 셀에 안 번진다.

테넌트별 백업·복원

한 테넌트 상태만 백업에서 복원, 타 테넌트 영향 없음.

규제 준수

GDPR·CCPA의 데이터 접근·삭제권을 샤드 단위 export·삭제로 단순화.

데이터 거주·점진 롤아웃

특정 관할권에 샤드 배치(데이터 거주법). 스키마 마이그레이션을 테넌트별 점진 적용해 위험↓.

멀티테넌시의 도전

테넌트 샤딩의 현실적 한계

  • 각 테넌트가 한 노드에 들어갈 만큼 작아야 한다. 한 테넌트가 너무 크면 그 안에서 또 샤딩해야 한다 → 확장성 샤딩 문제로 회귀.
  • 작은 테넌트가 아주 많으면 테넌트마다 샤드를 만드는 오버헤드가 크다. 묶으면 — 성장 시 테넌트를 옮기는 문제가 생긴다.
  • 여러 테넌트에 걸친 기능이 필요해지면, 여러 샤드를 조인해야 해 구현이 어려워진다.
키-값 데이터 샤딩

목표: 데이터와 부하를 균등하게

균등하면 10개 노드가 단일 노드의 10배 데이터·처리량을 (이론상) 감당한다. 노드 추가·제거 시 리밸런싱으로 다시 고르게 분산해야 한다.

치우침 (Skew)

일부 샤드에 데이터·질의가 몰리면 샤딩 효과가 크게 떨어진다. 극단적으로 모든 부하가 한 샤드에 몰려 9/10 노드가 놀 수 있다.

핫스팟 · 핫키

부하가 과도하게 높은 샤드 = 핫 샤드(hot spot). 한 키가 특히 부하 높으면(소셜의 셀럽) = 핫 키(hot key).

파티션 키를 입력받아 어느 샤드인지 알려주는 알고리즘이 필요하다. 키-값 저장소에선 보통 키(또는 키의 첫 부분), 관계형에선 테이블의 한 컬럼(꼭 PK일 필요는 없음)이 파티션 키다. 이 알고리즘은 리밸런싱이 가능해야 한다.

키 범위 샤딩

백과사전처럼 연속 범위 할당

최소~최대 키의 연속 범위를 각 샤드에 할당. 제목으로 찾으면 어느 권(샤드)에 있는지 바로 안다. 범위가 균등하지 않을 수 있어(A·B 한 권, T~Z 한 권) 경계가 데이터에 맞게 적응해야 한다.

  • 샤드 내 키는 정렬 저장(B-트리·SSTable) → 범위 스캔이 쉽다.
  • 인접 키에 쓰기가 몰리면 핫 샤드 — 타임스탬프 키면 이번 달 샤드에 쓰기 집중.
권 1
A · B
권 2
C · D · E
권 3
F ~ K
권 12
T·U·V
W·X·Y·Z
핫스팟 회피: 키 앞에 센서 ID를 붙여 정렬을 센서 ID → 타임스탬프 순으로. 쓰기가 분산되지만, 여러 센서를 시간 범위로 조회하려면 센서마다 별도 범위 질의가 필요해진다.
Vitess(수동) · Bigtable·HBase·MongoDB·CockroachDB·FoundationDB(자동) · YugabyteDB(둘 다)
키 범위 리밸런싱

분할(split)과 병합(merge)

  • 사전 분할(pre-splitting): 빈 DB에 초기 샤드 집합 구성. 키 분포를 미리 알아야 함.
  • 데이터·처리량이 늘면 샤드를 둘 이상으로 분할, 여러 노드에 분산. 많이 삭제되면 작은 인접 샤드를 병합.
  • 분할 트리거: 설정 크기 도달(HBase 기본 10GB) 또는 쓰기 처리량이 임계 초과 → 데이터가 적어도 핫 샤드는 분할.
분할은 비싸다: 모든 데이터를 새 파일로 재작성해야 한다(LSM의 컴팩션과 유사). 분할이 필요한 샤드는 보통 고부하 상태라, 분할 비용이 부하를 가중시켜 과부하 위험이 있다.

샤드 수가 데이터 양에 적응한다 — 데이터가 적으면 샤드 수도 적어 오버헤드↓, 많으면 개별 샤드 크기를 상한으로 제한.

키 해시 샤딩

먼저 해싱한 뒤 매핑

키가 가까이 모일 필요가 없다면(예: 테넌트 ID), 파티션 키를 해싱해 샤드에 매핑. 좋은 해시 함수는 치우친 데이터를 균등 분포로 만든다 — 비슷한 입력도 해시는 고르게 퍼진다.

암호학적으로 강할 필요는 없다 — MongoDB는 MD5, Cassandra·ScyllaDB는 Murmur3.
주의: 언어 내장 해시는 부적합할 수 있다 — Java의 Object.hashCode(), Ruby의 Object#hash는 프로세스마다 같은 키에 다른 해시값을 줄 수 있어 샤딩에 못 쓴다.

대가

키의 정렬이 깨져 범위 질의가 비효율적이 된다.

해시 mod N의 함정

hash(key) % N 은 왜 나쁜가

노드 수 N으로 나눈 나머지로 노드를 정하면 계산은 쉽다. 그러나 N이 바뀌면 대부분의 키가 이동한다.

3 → 4 노드 :
hash 3 → 노드3, hash 6 → 노드2,
hash 9 → 노드1 …

불필요한 데이터 이동이 너무 많다. 최소한의 데이터만 이동하는 방식이 필요하다.

고정 샤드 수

노드보다 훨씬 많은 샤드를 미리

10노드 클러스터를 처음부터 1,000 샤드로 나눠 노드당 100개 할당. 키는 hash(key) % 1000 샤드에. 노드 추가 시 기존 노드의 일부 샤드만 새 노드로 통째로 이동 — 샤드 분할보다 싸다.

  • 샤드 수·키→샤드 매핑은 불변, 샤드→노드 할당만 변한다.
  • 약한 하드웨어엔 적은 샤드, 강한 노드엔 많은 샤드 할당 가능.
한계: 처음에 샤드 수를 잘 추정해야 한다. 노드 수가 샤드 수를 넘어설 수 없다. 데이터 규모가 크게 변하면 — 너무 크면 리밸런싱·복구가 비싸고, 너무 작으면 오버헤드가 크다. 샤드 수가 잘못되면 비싼 리샤딩(각 샤드 분할·재작성, 다운타임 가능)이 필요.
Citus · Riak · Elasticsearch · Couchbase. 샤드 수는 보통 여러 약수를 갖는 수로(2의 거듭제곱일 필요 없음).
해시 범위 샤딩

키 대신 해시값의 범위를 할당

필요한 샤드 수를 미리 못 정하면, 각 샤드가 해시값의 범위를 갖게 한다(키 범위 + 해시 결합). 16비트 해시면 0~65,535를 샤드별로 분할 — 비슷한 키도 해시는 고르게 퍼진다.

  • 샤드가 커지면 필요할 때 분할 → 샤드 수가 데이터 양에 적응.
  • 파티션 키 범위 질의는 비효율(키가 모든 샤드에 흩어짐). 단, 복합 키에서 첫 컬럼이 파티션 키면 나머지 컬럼 범위 질의는 가능.
YugabyteDB·DynamoDB 사용, MongoDB 옵션. Cassandra·ScyllaDB는 변형 사용.

일관 해싱

Consistent Hashing

두 성질을 만족하는 해시 함수:

  • 각 샤드에 매핑되는 키 수가 대략 균등.
  • 샤드 수가 바뀔 때 가능한 한 적은 키만 이동.
여기서 "일관(consistent)"은 복제 일관성·ACID 일관성과 무관하다 — 키가 가능하면 같은 샤드에 머무는 성향을 뜻한다.

변종들

Cassandra·ScyllaDB의 방식은 원조 일관 해싱과 유사. 그 외 최고 무작위 가중치(랑데부 해싱), 점프 일관 해싱 등. 이들은 기존 샤드를 쪼개는 대신, 신규 노드에 여러 노드에 흩어져 있던 개별 키를 할당한다. 어느 쪽이 나은지는 애플리케이션에 달렸다.

치우친 워크로드 · 핫스팟 완화

균등 분포 ≠ 균등 부하

일관 해싱이 키를 고르게 분산해도 실제 부하는 고르지 않을 수 있다. 셀럽의 게시물 하나가 같은 키로 폭발적 읽기·쓰기를 부른다.

시스템 레벨

키(또는 해시) 범위로 샤드를 정의하면 핫 키를 단독 샤드에, 심지어 전용 머신에 둘 수 있다. 클라우드는 자동 대응 — 아마존의 열 관리(heat management)·적응 용량(adaptive capacity).

앱 레벨

핫 키 앞뒤에 난수 부여 — 두 자리만 붙여도 쓰기를 100개 키로 분산. 단, 읽기는 100개 키를 모두 읽어 합쳐야 하고(쓰기만 분산), 어느 키를 분할했는지 추적하는 부기가 필요.

시간에 따라 부하가 변한다 — 바이럴 게시물은 며칠 뜨겁다 식는다. 어떤 키는 쓰기, 어떤 키는 읽기에 뜨거워 서로 다른 전략이 필요하다.
자동 vs 수동 리밸런싱

사람을 루프에 둘 것인가

완전 자동

운영 부담↓, 워크로드 변화에 오토스케일 가능(DynamoDB는 수 분 내 샤드 추가·제거). 하지만 예측 불가할 수 있다 — 리밸런싱은 비싼 연산이라 네트워크·노드를 과부하시킬 수 있다.

자동의 위험

자동 장애 감지와 결합하면 위험. 과부하로 느려진 노드를 죽었다 판단 → 부하를 옮김 → 다른 노드 과부하 → 연쇄 장애(cascading failure)로 번질 수 있다.

중간 지대도 있다 — Couchbase·Riak은 자동으로 샤드 할당을 제안하되 관리자가 승인해야 적용. 수동 리밸런싱은 느리지만 운영상 사고를 막고, 알려진 트래픽 급증(사이버 먼데이·월드컵 티켓)에 선제적으로 대비할 수 있다.
요청 라우팅

어느 노드에 연결해야 하나?

서비스 디스커버리와 유사하지만, 키 요청은 그 샤드의 복제본 노드만 처리할 수 있다는 점이 다르다. 세 가지 접근:

① 아무 노드

라운드로빈 LB로 아무 노드에. 그 노드가 해당 샤드를 가지면 직접 처리, 아니면 적절한 노드로 전달 후 응답을 중계.

② 라우팅 계층

모든 요청을 먼저 라우팅 계층으로. 요청을 처리하진 않고, 샤드를 아는 LB 역할로 전달만 한다.

③ 샤드-인지 클라이언트

클라이언트가 샤드→노드 할당을 알아 직접 적절한 노드에 연결, 중개자 없음.

핵심 난제: 누가 샤드 배치를 결정하나(코디네이터의 내결함성·스플릿 브레인 방지)? 라우팅 주체는 할당 변경을 어떻게 학습하나? 샤드 이전 중 옛 노드로 향하던 요청은?
코디네이션 서비스

ZooKeeper · etcd로 할당 추적

많은 분산 시스템이 별도 코디네이션 서비스로 샤드 할당을 추적한다. 합의 알고리즘(10장)으로 내결함성·스플릿 브레인 방지. 각 노드가 등록하면 ZooKeeper가 권위 있는 샤드→노드 매핑을 유지하고, 변경 시 라우팅 계층에 알린다.

HBase·SolrCloud · ZooKeeperKubernetes · etcdMongoDB · config server + mongosKafka·TiDB·ScyllaDB · 내장 Raft
Riak는 다른 길: 노드 간 가십 프로토콜로 클러스터 상태 변화를 전파. 합의보다 약한 일관성이라 스플릿 브레인이 가능하지만, 리더리스 DB는 어차피 약한 일관성을 견디므로 용인한다.

클라이언트가 연결할 IP는 샤드 할당보다 덜 자주 바뀌므로, 이 목적엔 보통 DNS로 충분하다.

샤딩과 보조 인덱스

보조 인덱스는 샤드에 깔끔히 안 맞는다

지금까진 클라이언트가 파티션 키를 안다고 가정했다. 보조 인덱스는 레코드를 유일 식별하는 게 아니라 특정 값의 출현을 검색한다 — "사용자 123의 모든 행동", "빨간 차 전부". 전문 검색 엔진(Solr·Elasticsearch)의 존재 이유이기도 하다.

로컬 인덱스

각 샤드가 자기 레코드만 인덱싱. 쓰기는 한 샤드만, 읽기는 모든 샤드 조회.

전역 인덱스

모든 샤드 데이터를 덮되, 인덱스 자체를 인덱싱 값으로 별도 샤딩.

로컬 보조 인덱스

문서-파티션 인덱스

중고차 사이트: ID로 샤딩(0~499 샤드0, 500~999 샤드1). color·make 인덱스를 각 샤드가 자기 레코드에 대해서만 유지. 빨간 차가 추가되면 그 샤드가 color:red 포스팅 리스트에 ID 추가.

  • 파티션 키를 알면 해당 샤드만 검색. 일부 결과만 원하면 아무 샤드에나.
  • 전체 결과를 원하고 파티션 키를 모르면 모든 샤드에 질의 후 결합 — 꼬리 지연 증폭, 확장성 제한.
MongoDB · Riak · Cassandra · Elasticsearch · SolrCloud · VoltDB
전역 보조 인덱스

용어-파티션 인덱스

모든 샤드를 덮는 전역 인덱스를 만들되, 한 노드에 두면 병목이 되므로 인덱싱 값으로 다시 샤딩. color a~r은 샤드0, s~z는 샤드1. 용어(term)를 파티션 키로 써, 찾는 값이 어느 샤드인지 안다.

  • 단일 조건 질의(color=red)는 한 샤드만 읽어 포스팅 리스트를 얻는다.
  • 여러 조건의 AND는 용어들이 다른 샤드에 → 긴 포스팅 리스트의 교집합 계산이 네트워크상 느릴 수 있다.
  • 쓰기가 복잡 — 한 레코드가 인덱스의 여러 샤드에 영향. 분산 트랜잭션으로 원자 갱신하거나(8장), 비동기 갱신(DynamoDB는 전역 인덱스 읽기가 stale 가능).
CockroachDB · TiDB · YugabyteDB · DynamoDB(로컬·전역 둘 다)
7장 요약

샤딩 — 한눈에

목표는 데이터·질의 부하를 머신에 균등 분산해 핫스팟을 피하는 것. 데이터에 맞는 스킴을 고르고, 노드 추가·제거 시 리밸런싱한다.

두 가지 샤딩 방식
  • 키 범위 샤딩 — 정렬된 키 범위를 소유. 범위 질의 효율적이나 인접 키 핫스팟 위험. 너무 커지면 둘로 분할.
  • 해시 샤딩 — 해시값 범위를 소유. 정렬 파괴로 범위 질의 비효율, 부하는 더 균등. 보통 고정 수 샤드를 만들어 노드 간 통째 이동.
키 첫 부분을 파티션 키로, 나머지로 샤드 내 정렬 → 같은 파티션 키 안에서 범위 질의 유지.
보조 인덱스의 두 방식
  • 로컬 — 같은 샤드에 인덱스 저장. 쓰기는 한 샤드, 읽기는 모든 샤드.
  • 전역 — 인덱싱 값 기준 별도 샤딩. 쓰기는 여러 인덱스 샤드 갱신, 읽기(포스팅 리스트)는 한 샤드에서.
각 샤드는 대체로 독립 동작 — 그것이 확장의 비결이다. 그러나 여러 샤드에 쓰기가 필요할 때(한 샤드 성공·다른 샤드 실패 시?)가 문제다 — 다음 장들에서 다룬다.
