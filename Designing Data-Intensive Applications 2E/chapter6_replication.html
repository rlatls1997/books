복제
Replication

네트워크로 연결된 여러 머신에 동일한 데이터의 복사본을 유지하는 것. 변경되지 않는 데이터의 복제는 쉽다 — 모든 어려움은 변경의 전파를 다루는 데 있다.

단일 리더
한 노드만 쓰기를 받고 변경을 흘려보냄
다중 리더
여러 노드가 쓰기를 받고 서로 동기화
리더리스
리더 없이 여러 노드에 직접 읽고 씀
왜 복제하는가

복제의 세 가지 목적

Latency

지리적 근접성

데이터를 사용자 가까이에 두어 접근 지연을 줄인다.

Availability

가용성·내구성

일부 부품이 고장 나도 시스템이 계속 동작하게 한다.

Throughput

읽기 확장

읽기 질의를 처리하는 머신 수를 늘려 읽기 처리량을 높인다.

이 장에서는 데이터셋이 각 머신에 전체가 들어갈 만큼 작다고 가정한다. 너무 커서 한 머신에 안 들어가는 경우의 샤딩(파티셔닝)은 7장에서 다룬다. 복제 원리는 1970년대 이후 거의 바뀌지 않았다 — 네트워크의 근본 제약이 그대로이기 때문이다.
백업 vs 복제

복제가 있어도 백업은 필요한가?

그렇다. 목적이 다르기 때문이다.

  • 복제는 한 노드의 쓰기를 다른 노드에 빠르게 반영한다.
  • 백업은 과거 스냅숏을 저장해 시간을 되돌릴 수 있게 한다.
  • 실수로 데이터를 삭제하면 삭제도 복제본에 전파되므로, 복원하려면 백업이 필요하다.

상호 보완

백업은 복제 설정 과정의 일부가 되기도 하고, 반대로 복제 로그를 아카이빙하는 것이 백업 과정의 일부가 되기도 한다.

오브젝트 스토리지

현재 상태는 기본 스토리지에 두고, 자주 접근하지 않는 과거 데이터 백업은 저렴한 오브젝트 스토어에 보관하는 것이 더 경제적일 수 있다.

단일 리더 복제 · Single-Leader

리더-기반(주-부, active/passive) 복제

리더(primary): 모든 쓰기 요청을 받아 먼저 자기 로컬 스토리지에 쓴다.
팔로워(read replica): 리더가 보내는 복제 로그/변경 스트림을 받아 리더와 동일한 순서로 적용한다.
읽기는 리더나 팔로워 어디서든 가능하지만, 쓰기는 오직 리더만 받는다.
PostgreSQLMySQLOracle Data GuardMongoDBKafkaRaft (CockroachDB·etcd·TiDB)

변경 스트림이 리더 → 팔로워로 흐른다

동기 vs 비동기

언제 "성공"을 클라이언트에게 알리는가

동기식 (Synchronous)

리더가 팔로워의 수신 확인을 기다린 뒤 성공을 보고한다.

장점 — 팔로워가 항상 최신 복사본을 보장. 리더가 죽어도 데이터가 살아있다.
단점 — 팔로워가 응답하지 않으면 쓰기 전체가 막힌다.

비동기식 (Asynchronous)

리더가 메시지만 보내고 응답을 기다리지 않는다.

장점 — 팔로워가 뒤처져도 리더는 계속 쓰기를 처리.
단점 — 리더가 복구 불가로 죽으면 미복제 쓰기는 유실. 확정된 쓰기도 내구성을 보장 못 한다.

모든 팔로워를 동기식으로 두는 건 비현실적이다 — 단 한 노드만 죽어도 전체가 멈춘다. 보통 한 팔로워만 동기식, 나머지는 비동기식(semisynchronous)으로 두어 최소 두 노드(리더+동기 팔로워)에 최신본을 보장한다.
복제 강도의 스펙트럼

완전 동기 ↔ 정족수 ↔ 완전 비동기

반동기 (Semisync)

하나는 동기, 나머지는 비동기. 동기 팔로워가 느려지면 비동기 중 하나를 동기로 승격.

다수결 정족수

5개 중 3개 등 과반을 동기로 갱신. 합의 프로토콜·자동 리더 선출 시스템에서 흔히 쓰인다. (10장)

완전 비동기

내구성을 약화하지만, 팔로워가 많거나 지리적으로 분산된 경우 널리 쓰인다.

정상 동작 시 복제는 보통 1초 미만으로 빠르다. 그러나 보장은 없다 — 팔로워가 복구 중이거나, 시스템이 한계 부하이거나, 네트워크 문제가 있으면 수 분 이상 뒤처질 수 있다.

새 팔로워 설정

다운타임 없이 새 팔로워 추가하기

파일을 그냥 복사하면 안 된다 — 데이터가 계속 변하므로 시점이 뒤섞인다. 잠그면 가용성을 해친다. 대신:

일관된 스냅숏을 (가능하면 잠그지 않고) 리더에서 특정 시점에 찍는다.
스냅숏을 새 팔로워 노드로 복사한다.
팔로워가 리더에 접속해, 스냅숏 이후의 모든 변경을 요청한다. 이때 스냅숏은 복제 로그의 정확한 위치(PostgreSQL의 LSN, MySQL의 binlog 좌표·GTID)와 연결돼 있어야 한다.
밀린 변경을 모두 처리하면 따라잡았다(caught up)고 하고, 이후 실시간 변경을 계속 수신한다.
복제 로그를 주기적 스냅숏과 함께 오브젝트 스토어에 아카이빙하면 백업·재해 복구가 되고, 1·2단계를 스토어에서 다운로드로 수행할 수 있다. (PostgreSQL의 WAL-G, SQLite의 Litestream)
오브젝트 스토리지 기반 DB

S3·GCS·Blob을 라이브 질의에 쓰기

이점
  • 저렴하다 — 덜 쓰이는 데이터를 싼 고지연 스토리지에, 작업 집합은 메모리·SSD·NVMe에.
  • 멀티존·멀티리전 복제와 매우 높은 내구성, 존 간 네트워크 요금 회피.
  • 조건부 쓰기(CAS)로 트랜잭션·리더 선출 구현.
  • Parquet·Iceberg 같은 개방 포맷으로 데이터 통합 단순화.
트레이드오프
  • 로컬 디스크·EBS보다 읽기/쓰기 지연이 훨씬 높다.
  • API 호출당 과금 → 배치 처리 강제 → 지연 증가.
  • 객체는 보통 불변 → 큰 객체의 랜덤 쓰기는 매우 비싸다.
  • 표준 파일시스템 인터페이스 부재(FUSE도 POSIX 일부 미지원).
제로 디스크 아키텍처(ZDA) — 모든 데이터를 오브젝트 스토리지에 영속화하고 디스크·메모리는 캐싱에만. 노드가 영속 상태를 갖지 않아 운영이 극적으로 단순해진다. WarpStream · Bufstream · Redpanda Serverless · SlateDB · 거의 모든 클라우드 데이터 웨어하우스
노드 장애 처리 ①

팔로워 장애 — 따라잡기 복구

팔로워는 로컬 로그에 받은 변경을 기록한다. 크래시·네트워크 단절 후 재시작하면:

  • 로그에서 마지막으로 처리한 트랜잭션을 안다.
  • 리더에 접속해 단절 동안의 변경을 요청·적용하면 따라잡는다.
리더의 딜레마: 팔로워가 오래 사라지면 — 로그를 보관하다 디스크가 차거나(보관), 미확인 로그를 삭제해 팔로워가 백업에서 복원해야 하거나(삭제) 둘 중 하나를 택해야 한다.

처리량이 높거나 오래 오프라인이었으면 밀린 쓰기가 많아, 복구 중 팔로워와 리더 모두 부하가 높아진다.

노드 장애 처리 ②

리더 장애 — 페일오버(failover)

팔로워 하나를 새 리더로 승격하고, 클라이언트를 재구성하고, 다른 팔로워들이 새 리더를 따르게 한다. 자동 페일오버는 보통:

리더 사망 판단 — 완벽한 방법이 없어 대부분 타임아웃 사용(예: 30초 무응답이면 사망 간주).
새 리더 선택 — 선거 또는 컨트롤러 노드 지정. 데이터 유실을 최소화하려면 가장 최신인 복제본이 최선. 이것은 합의 문제(10장).
시스템 재구성 — 클라이언트가 새 리더로 쓰기를 보내게 한다. 옛 리더가 돌아오면 자신이 리더라 착각할 수 있으니, 팔로워로 강등시켜야 한다.
가장 중요한 것: 최신 팔로워를 새 리더로 고를 것. 동기/반동기면 리더가 기다리던 그 팔로워, 비동기면 LSN이 가장 높은 팔로워. 1초 미만 유실은 견딜 만하지만 며칠 뒤처진 복제본을 고르면 재앙이다.
페일오버의 함정

잘못될 수 있는 모든 것

쓰기 유실

비동기 시 새 리더가 옛 쓰기를 못 받았을 수 있다. 옛 리더 복귀 시 충돌하는 쓰기가 생기면, 보통 옛 리더의 미복제 쓰기를 버린다 → 확정된 줄 알았던 데이터가 사라진다.

외부 시스템 불일치

GitHub 사건: 뒤처진 MySQL 팔로워가 승격되며 오토인크리먼트 카운터가 뒤처져 PK를 재사용 → Redis와 불일치, 사생활 정보가 잘못된 사용자에게 노출.

스플릿 브레인

두 노드가 모두 자신을 리더라 믿는 상황. 둘 다 쓰기를 받고 충돌 해결이 없으면 데이터 유실·손상. 한쪽을 끄는 안전장치도 잘못 설계하면 양쪽이 다 꺼질 수 있다.

타임아웃 설정

너무 길면 복구 지연, 너무 짧으면 불필요한 페일오버(부하 급증·네트워크 글리치). 이미 부하가 높을 때의 불필요한 페일오버는 상황을 악화시킨다.

옛 리더를 제한·차단해 스플릿 브레인을 막는 것을 펜싱(fencing)이라 한다. 쉬운 해법은 없어, 일부 운영팀은 자동 페일오버를 지원해도 수동 페일오버를 선호한다.
복제 로그 구현 ①

구문 기반 (Statement-based)

리더가 실행한 모든 쓰기 구문(INSERT·UPDATE·DELETE)을 로그로 보내고, 팔로워가 SQL을 그대로 파싱·실행한다.

컴팩트하지만 깨지기 쉽다. 비결정성이 있으면 MySQL 5.1+는 기본적으로 행 기반으로 전환. VoltDB는 트랜잭션을 결정적으로 강제해 안전하게 사용. (상태 기계 복제)
깨지는 경우
  • 비결정 함수 NOW()·RAND() → 복제본마다 다른 값.
  • 오토인크리먼트·기존 데이터 의존(UPDATE ... WHERE) → 정확히 같은 순서로 실행돼야 함.
  • 부수효과(트리거·저장 프로시저·UDF) → 복제본마다 다른 효과.
복제 로그 구현 ②

WAL 배송 (Write-Ahead Log Shipping)

B-트리의 크래시 복구용 WAL을 그대로 네트워크로 보낸다. 팔로워가 처리하면 리더와 동일한 파일을 만든다. PostgreSQL · Oracle

단점: 스토리지 엔진과 강결합

WAL은 어떤 디스크 블록의 어떤 바이트가 바뀌었는지 같은 매우 저수준 데이터다. 저장 포맷이 바뀌면 리더·팔로워가 다른 버전의 DB를 못 돌린다.

운영 영향: 버전 불일치를 허용하지 않으면 무중단 업그레이드(팔로워 먼저 올리고 페일오버)가 불가능 → 업그레이드에 다운타임 필요.
복제 로그 구현 ③

논리적(행 기반) 로그

복제용과 스토리지용 로그 포맷을 분리. 행 단위로 쓰기를 기술하는 논리 로그.

INSERT

삽입된 행의 모든 컬럼 새 값을 기록.

DELETE

행을 유일하게 식별할 정보(보통 PK, 없으면 모든 컬럼의 옛 값).

UPDATE

식별 정보 + 변경된(또는 모든) 컬럼의 새 값.

호환성: 엔진 내부와 분리돼 하위 호환 유지 쉬움 → 리더·팔로워가 다른 DB 버전 가능 → 최소 다운타임 업그레이드. (MySQL binlog, PostgreSQL 논리 복제)
외부 활용: 외부 앱이 파싱하기 쉬워, DB 내용을 데이터 웨어하우스·맞춤 인덱스·캐시로 보낼 수 있다 → 변경 데이터 캡처(CDC) (12장).
복제 지연의 문제

결과적 일관성 (Eventual Consistency)

읽기 대부분 + 쓰기 소량인 워크로드는 매력적인 선택지가 있다: 팔로워를 많이 만들어 읽기를 분산하는 읽기 확장(read scaling). 하지만 이는 현실적으로 비동기 복제에서만 동작한다.

비동기 팔로워에서 읽으면 오래된 정보를 볼 수 있다. 리더와 팔로워에 같은 질의를 동시에 하면 다른 결과가 나온다.

쓰기를 멈추고 기다리면 팔로워는 결국 따라잡는다 — 그래서 결과적 일관성. "결국(eventually)"은 의도적으로 모호하다. 정상 시엔 1초 미만이지만, 한계 부하·네트워크 문제 시 수 초~수 분으로 커진다.
용어는 Douglas Terry가 만들고 Werner Vogels가 대중화. NoSQL만이 아니라 비동기 복제 관계형 DB의 팔로워도 동일한 특성을 갖는다.

지연이 커질 때 나타나는 세 가지 대표적 이상 현상을 살펴본다.

이상 현상 ①

자기 쓰기 읽기
(Read-after-write)

사용자가 데이터를 제출한 직후 보면, 새 데이터가 아직 복제본에 도달하지 않아 "내 입력이 사라졌다"고 느낀다.

읽기-후-쓰기 일관성: 사용자가 페이지를 새로고침하면 자기가 제출한 변경은 항상 보이도록 보장. 타인의 변경은 나중에 보여도 된다.
읽기-후-쓰기 구현 기법

어떻게 보장하는가

① 수정 가능 데이터는 리더에서

사용자가 수정했을 만한 것은 리더(또는 동기 팔로워)에서 읽기. 예: 소셜 프로필은 본인만 수정 → 내 프로필은 항상 리더, 타인 것은 팔로워.

② 시간 기반 라우팅

대부분이 수정 가능하면, 마지막 업데이트 후 1분간 모든 읽기를 리더로. 또는 1분 이상 뒤처진 팔로워에는 질의 금지.

③ 타임스탬프 기억

클라이언트가 최근 쓰기의 타임스탬프(LSN·논리 클록·실시간 시계)를 기억 → 그 시점 이상 반영된 복제본만 읽기. 부족하면 다른 복제본으로 보내거나 대기.

④ 리전 라우팅

복제본이 리전에 분산돼 있으면, 리더가 있어야 하는 요청은 리더가 있는 리전으로 라우팅해야 한다.

기기 간(cross-device) 일관성: 한 기기에서 입력한 걸 다른 기기에서도 봐야 한다면 — 타임스탬프 메타데이터를 중앙화해야 하고, 다른 기기의 연결이 같은 리전으로 라우팅되리란 보장이 없다.
개념 정리

리전과 가용 영역(AZ)

  • 리전(Region) = 한 지리적 위치의 데이터센터들.
  • 가용 영역(Zone/AZ) = 자체 전원·냉각을 가진 개별 데이터센터. 한 리전 = 여러 존.
  • 같은 리전의 존들은 초고속으로 연결돼, 대부분의 분산 시스템이 단일 존처럼 여러 존에 걸쳐 동작 가능.
존 장애는 멀티존 구성으로 견딜 수 있지만, 한 리전의 모든 존이 죽는 리전 장애는 막지 못한다. 리전 장애를 견디려면 멀티리전 배포가 필요하며, 이는 더 높은 지연·낮은 처리량·증가한 네트워크 비용을 부른다.
이상 현상 ②

단조 읽기
(Monotonic Reads)

사용자가 여러 복제본에서 읽으면 시간이 거꾸로 흐르는 것처럼 보일 수 있다. 먼저 지연 적은 팔로워에서 댓글을 봤다가, 지연 큰 팔로워에서 그 댓글이 사라진다.

단조 읽기: 결과적 일관성보다 강하고 강한 일관성보다 약한 보장. 한 사용자가 연속으로 읽을 때 더 오래된 데이터로 되돌아가지 않음을 보장.

구현법

각 사용자가 항상 같은 복제본에서 읽게 한다 — 예: 사용자 ID의 해시로 복제본 선택(무작위가 아니라). 단, 그 복제본이 죽으면 재라우팅 필요.

이상 현상 ③

일관된 접두 읽기
(Consistent Prefix)

인과성 위반. Mrs. Cake의 답("보통 10초")이 Mr. Poons의 질문("얼마나 미래를 보세요?")보다 먼저 도착하면 — 대답이 질문보다 먼저 보인다.

일관된 접두 읽기: 쓰기가 특정 순서로 일어났다면, 읽는 사람도 같은 순서로 본다.

샤딩 DB의 문제

여러 샤드가 독립 동작해 전역 쓰기 순서가 없을 때 발생. 어떤 부분은 옛 상태, 어떤 부분은 새 상태로 보인다.

해법: 인과적으로 관련된 쓰기를 같은 샤드에 쓰기. 또는 인과 의존성을 명시적으로 추적(happens-before 관계, 후술).

복제 지연의 해법

앱이 아니라 DB가 보장하게 하라

지연이 수 분~수 시간으로 커지면 어떻게 되는가? "문제없음"이면 좋지만, 사용자 경험을 해친다면 더 강한 보장이 필요하다. 비동기를 동기인 척하는 것은 재앙의 씨앗이다.

강한 일관성 + ACID

선형성(linearizability, 10장)과 ACID 트랜잭션(8장)을 제공하는 DB를 고르면, 복제의 난제를 대부분 무시하고 단일 노드처럼 다룰 수 있다.

NewSQL

2010년대 초 NoSQL은 이런 기능이 확장성을 제한한다 주장했으나, 이후 강한 일관성·트랜잭션 + 분산 DB의 내결함성·확장성을 함께 제공하는 NewSQL이 등장.

그래도 더 약한 일관성의 복제를 택하는 이유: 네트워크 단절에 더 강하고, 트랜잭션 시스템보다 오버헤드가 낮다.
다중 리더 복제 · Multi-Leader

하나 이상의 노드가 쓰기를 받게 하기

단일 리더의 큰 약점: 모든 쓰기가 한 리더를 거쳐야 한다. 리더에 연결 못 하면 쓸 수 없다. 자연스러운 확장 — 여러 노드가 쓰기를 받고 각자 다른 노드로 전파(active/active, 양방향 복제). 각 리더는 동시에 다른 리더의 팔로워다.

동기식 다중 리더는 사실상 단일 리더와 같다 — A→B 동기 복제 중 네트워크가 끊기면 A에 쓸 수 없으니, B를 리더로 두는 것과 다름없다.
이 절은 비동기식 다중 리더에 집중한다. 어떤 리더든, 다른 리더와의 연결이 끊겨도 쓰기를 처리할 수 있다.
지리적 분산 운영

멀티리전: 단일 리더 vs 다중 리더

관점단일 리더다중 리더
성능모든 쓰기가 리더 리전으로 인터넷을 건너야 함 → 큰 지연.로컬 리전에서 처리 후 비동기 전파 → 리전 간 지연을 사용자에게 숨김.
리전 장애 내성리더 리전 장애 시 다른 리전 팔로워로 페일오버.각 리전이 독립 운영, 복귀 시 따라잡음.
네트워크 문제 내성리전 간 링크에 매우 민감 — 요청·응답을 그 링크로 기다려야 함.일시 단절 중에도 각 리전 리더가 독립적으로 쓰기 처리.
일관성직렬성 등 강한 일관성 가능.훨씬 약함 — 계좌 음수 방지·유저명 유일성 보장 불가. 분산 시스템의 근본 한계.
다중 리더는 종종 후방 이식된 기능이라 오토인크리먼트 키·트리거·무결성 제약과 미묘하게 충돌한다. 가능하면 피해야 할 위험한 영역으로 여겨진다. 단일 리전 내에선 복잡성 대비 이점이 적어 잘 쓰지 않는다.
다중 리더 토폴로지

쓰기가 전파되는 통신 경로

원형 (Circular)

각 노드가 한 노드에서 받아 다음으로 전달.

별형 (Star)

루트 노드가 모두에게 전달. 트리로 일반화 가능.

전대전 (All-to-all)

모든 리더가 다른 모든 리더에게 전달.

무한 복제 루프 방지: 각 노드에 고유 식별자를 주고, 쓰기에 거쳐온 노드들의 식별자를 태깅. 자기 식별자가 붙은 변경은 무시한다.
토폴로지의 문제

순서가 뒤바뀐 쓰기

  • 원형·별형: 한 노드만 죽어도 다른 노드 간 메시지 흐름이 끊긴다(수동 재구성 필요).
  • 전대전: 내결함성은 좋지만, 링크 속도 차로 메시지가 추월당할 수 있다 — INSERT보다 UPDATE가 먼저 도착.
단순 타임스탬프로는 부족하다(시계가 충분히 동기화되지 않음). 버전 벡터(version vectors)로 순서를 잡아야 한다. 많은 다중 리더 시스템이 이를 제대로 안 해 취약 — 문서를 꼼꼼히 읽고 철저히 테스트하라.
싱크 엔진 & 로컬-퍼스트

오프라인에서도 동작하는 앱

캘린더·실시간 협업 앱(Google Docs·Figma·Linear)은 다중 리더의 극단적 형태다. 각 기기/탭이 로컬 복제본 = 리더이고, 기기 간 비동기 다중 리더 복제(sync)가 일어난다. 지연은 인터넷 접속 시점에 따라 수 시간일 수도 있다.

싱크 엔진

변경을 잡아 온라인이면 즉시 전송, 오프라인이면 저장 후 전송. 협업자 변경을 받아 로컬에 병합·UI 갱신.

오프라인-퍼스트

오프라인에서 편집을 계속할 수 있는 앱. 오프라인 = 매우 큰 네트워크 지연.

로컬-퍼스트

개발사가 온라인 서비스를 모두 닫아도 동작. 개방형 싱크 프로토콜로 달성(예: Git + GitHub/GitLab).

싱크 엔진의 장단점

클라이언트에 영속 상태를 둔다

  • 로컬 데이터로 UI가 훨씬 빠르게 반응(60Hz에서 16ms 내 렌더 목표도).
  • 오프라인 작업 가능 — 별도 오프라인 모드 불필요.
  • 프로그래밍 모델 단순화 — 로컬 읽기·쓰기는 거의 실패하지 않아 선언적 스타일.
  • 리액티브 모델과 결합해 타인의 편집을 실시간 반영.
한계: 사용자가 필요한 데이터를 미리 다 받아 영속 저장해야 잘 동작한다. 한 사용자가 만든 파일은 괜찮지만, 이커머스 전체 카탈로그처럼 양이 크면 부적합.

1980년대 Lotus Notes가 선구. 오늘날: 독자 백엔드(Firestore·Realm·Ditto), 오픈소스 백엔드(PouchDB/CouchDB·Automerge·Yjs). 게임의 동급 개념은 넷코드(netcode).

충돌하는 쓰기 다루기

동시 쓰기 충돌

다중 리더의 가장 큰 문제. 사용자 1이 제목을 A→B로, 사용자 2가 A→C로 바꾸면 각자 로컬 리더엔 적용되지만, 비동기 복제 시 충돌이 감지된다. 단일 리더에선 없는 문제다.

"동시(concurrent)"란? 어느 쪽도 상대를 인지하지 못한 채 이뤄진 쓰기. 물리적으로 같은 시각인지는 중요치 않다(오프라인이면 시간차가 있을 수 있음). 핵심은 한 쪽이 다른 쪽이 적용된 상태에서 일어났는가이다.
충돌 해결 ①

충돌 회피
(Avoidance)

애초에 충돌이 안 생기게 한다. 특정 레코드의 모든 쓰기가 같은 리더를 거치게 보장하면 충돌이 없다.

사용자가 자기 데이터만 편집하는 앱에서, 한 사용자의 요청을 항상 같은 리전으로 라우팅 → 그 사용자 관점에선 사실상 단일 리더.

회피의 한계

지정 리더를 바꿔야 할 때(리전 장애·사용자 이동) 전환 중 쓰기가 일어나면 충돌이 생긴다. 리더 변경을 허용하면 회피는 깨진다.

ID 생성 예

두 리더 중 하나는 홀수, 하나는 짝수 ID만 생성 → 같은 ID 동시 할당을 막는다.

충돌 해결 ②

최종 쓰기 승리 (LWW)

각 쓰기에 타임스탬프를 붙여 가장 큰(최신) 값을 채택. 같으면 값을 비교(문자열은 사전순). 모든 복제본이 결국 일관된 상태가 되지만 — 데이터 유실이 대가다.

이름이 오해를 부른다: 동시 쓰기에서 "어느 게 최신인지"는 정의되지 않으므로, 사실상 무작위로 한 쓰기를 승자로 고르고 나머지를 조용히 버리는 것이다.

안전한 경우

유일 키로 삽입만 하고 갱신하지 않으면 LWW는 문제없다.

시계 의존

실시간 시계(Unix 타임스탬프)를 쓰면 시계가 빠른 노드의 값에 가려 나중 쓰기가 무시될 수 있다 → 논리 클록으로 해결.

충돌 해결 ③

수동 해결 & 형제(siblings)

충돌이 복제를 멈추게 할 순 없으니, DB는 동시에 쓰인 값들을 모두 저장(B와 C = 형제)하고, 다음 읽기에서 모두 반환한다. 앱이 자동으로(B/C로 연결) 또는 사용자에게 물어 해결. 예: CouchDB

문제들

API가 바뀐다(문자열 → 문자열 집합). 사용자에게 병합을 시키는 건 큰 부담. 부주의한 자동 병합은 놀라운 결과를 부른다.

아마존 장바구니 이상

충돌을 합집합으로 병합 → 한 기기에서 삭제한 상품이 다른 형제에 남아있어 되살아난다. 기기1이 Book 삭제, 기기2가 DVD 삭제했는데 병합 후 둘 다 부활.

여러 노드가 동시에 해결하면 해결 과정 자체가 새 충돌을 만들 수 있다 — 한 노드는 B/C, 다른 노드는 C/B → 병합 시 B/C/C/B 같은 결과.
충돌 해결 ④

자동 충돌 해결

동시 쓰기를 일관된 상태로 자동 병합. 같은 쓰기 집합을 처리한 모든 복제본이 도착 순서와 무관하게 같은 상태로 수렴 → 결과적 일관성 + 수렴 보장 = 강한 결과적 일관성(strong eventual consistency).

데이터 타입별 병합
  • 텍스트: 삽입·삭제를 추적해 모든 변경 보존. 같은 위치 동시 삽입은 결정적으로 정렬.
  • 컬렉션: 장바구니 이상 회피 — Book·DVD 삭제를 추적해 결과는 {Soap}.
.
  • 카운터(좋아요 수): 각 형제의 증감을 세어 정확히 합산 — 중복·누락 없음.
  • 키-값 맵: 같은 키는 다른 알고리즘으로 병합, 다른 키는 독립 처리.
한계: 리스트 5개 제한 같은 제약은 동시에 5개 초과 추가 시 일부를 버릴 수밖에 없다. 그래도 협업·로컬-퍼스트 앱에선 자동화가 최선인 경우가 많다.
자동 병합 알고리즘

CRDT vs OT

"ice"에 한쪽은 앞에 n("nice"), 다른 쪽은 끝에 !("ice!"). 둘 다 "nice!"로 병합 — 방식이 다르다.

Operational Transformation

OT

삽입/삭제 인덱스를 기록(n@0, !@3). 연산 교환 후, 이미 적용된 동시 연산을 고려해 인덱스를 변환 — !는 index 4로 보정해야 nic!e 오류 회피. 주로 실시간 텍스트 협업(Google Docs).

Conflict-free Replicated Data Type

CRDT

각 문자에 불변 고유 ID 부여(i=1A...). 삽입 시 새 문자 ID(4B)와 앞 문자 ID(3A)를 담아 인덱스 대신 ID로 위치 결정. 같은 위치 동시 삽입은 ID로 정렬 → 변환 없이 수렴. 분산 DB(Redis Enterprise·Riak·Azure Cosmos DB).

JSON 싱크 엔진은 CRDT(Automerge·Yjs)로도, OT(ShareDB)로도 구현 가능. 둘의 장점을 한 알고리즘에 결합하는 것도 가능.

리더리스 복제 · Leaderless

리더라는 개념을 버리다

어떤 복제본이든 클라이언트의 쓰기를 직접 받는다. 초기 복제 시스템 일부가 리더리스였으나 관계형 DB 시대에 잊혔다가, 2007년 아마존의 Dynamo로 다시 유행. Riak · Cassandra · ScyllaDB = Dynamo 스타일

코디네이터: 클라이언트가 여러 복제본에 직접 쓰거나, 코디네이터 노드가 대신 한다. 단, 리더와 달리 쓰기 순서를 강제하지 않는다 — 이 차이가 사용 방식에 큰 영향.
이름 주의: 논문의 원조 Dynamo는 외부 공개된 적 없다. 비슷한 이름의 DynamoDB는 완전히 다른 아키텍처 — Multi-Paxos 기반 단일 리더 복제다.
노드가 죽었을 때 쓰기

페일오버가 없다

모든 복제본이 동등하므로 페일오버 자체가 없다. 클라이언트가 3개 복제본에 병렬로 쓰고, 2개의 OK면 성공으로 간주. 죽은 복제본이 놓친 것은 무시.

읽기도 여러 노드에 병렬로 보낸다. 어떤 노드는 최신, 어떤 노드는 stale 값을 반환할 수 있다. 모든 값에 버전 번호·타임스탬프를 붙이고, 클라이언트는 가장 큰 버전을 채택한다.
놓친 쓰기 따라잡기

결국 모든 데이터가 모든 복제본에

읽기 복구
(Read repair)

병렬 읽기 중 stale 응답을 감지하면, 클라이언트가 최신값을 그 복제본에 다시 써준다. 자주 읽히는 값에 적합.

힌트 핸드오프
(Hinted handoff)

복제본이 죽으면 다른 복제본이 대신 쓰기를 힌트로 저장. 복귀하면 힌트를 전달 후 삭제. 읽히지 않는 값도 따라잡게 함.

안티 엔트로피
(Anti-entropy)

백그라운드 프로세스가 복제본 간 차이를 주기적으로 찾아 빠진 데이터를 복사. 순서 보장 없고 지연이 클 수 있다.

정족수 읽기·쓰기

w + r > n

n = 복제본 수, 쓰기는 w개의 확인 필요, 읽기는 r개에 질의. w + r > n이면 읽기·쓰기 노드 집합이 겹쳐 최신값을 보장한다.

n=3, w=2, r=2 → 1대 장애 허용
n=5, w=3, r=3 → 2대 장애 허용

보통 n은 홀수(3·5), w=r=(n+1)/2. 읽기 많고 쓰기 적으면 w=n, r=1도 가능(읽기 빠름, 단 한 노드 장애로 쓰기 전부 실패).

정족수 일관성의 한계

w + r > n 이어도 흔들리는 경우

  • 새 값 노드가 죽고 옛 값 복제본에서 복원되면 w 미만으로 떨어져 정족수 깨짐.
  • 리밸런싱 중 어느 노드가 n개 복제본인지 견해가 달라 읽기·쓰기 정족수가 안 겹칠 수 있다.
  • 읽기와 쓰기가 동시일 때 — 새 값을 볼 수도, 그 다음 읽기가 옛 값을 볼 수도.
  • 쓰기가 일부 성공·일부 실패해 전체 w 미만이면 롤백되지 않아, 실패 보고된 쓰기가 이후 읽기에 보일 수도.
  • 실시간 시계로 최신 판단 시(Cassandra·ScyllaDB), 시계 빠른 노드가 덮어쓰면 쓰기가 조용히 사라질 수 있다.
  • 두 쓰기가 동시면 복제본마다 처리 순서가 달라 충돌 발생.
정족수가 최신값을 보장하는 듯 보여도 실제론 단순하지 않다. Dynamo 스타일은 결과적 일관성을 견디는 용도에 최적화돼 있고, w·r은 stale 확률 조절 손잡이일 뿐 절대 보장이 아니다.
신선도 모니터링

"결국"을 정량화하기

stale 읽기를 견딜 수 있어도 복제 건강 상태는 알아야 한다. 크게 뒤처지면 알림으로 원인(네트워크·과부하)을 조사할 수 있어야.

리더 기반

쓰기가 같은 순서로 적용되고 각 노드가 복제 로그상 위치를 가지므로, 리더 위치 − 팔로워 위치로 복제 지연을 쉽게 측정.

리더리스

쓰기 적용에 고정 순서가 없어 모니터링이 어렵다. 힌트 핸드오프 수가 한 지표지만 해석이 까다롭다.

단일 리더 vs 리더리스 성능

실패와 정상을 구분하지 않는 회복탄력성

단일 리더 — 리더에서 읽기

최신 응답을 보장하나: 읽기 처리량이 리더 용량에 제한, 페일오버 대기, 리더 성능 문제에 매우 민감(과부하·자원 경합이 즉시 사용자에 영향).

리더리스 — 병렬 요청

페일오버가 없고 여러 복제본에 병렬 요청 → 한 복제본이 느려도 영향 작다(빠른 응답 사용 = 요청 헤징, 꼬리 지연 감소). 회색 장애·과부하에 특히 강하다.

리더리스의 성능 문제도 있다: 죽은 복제본 감지·핸드오프가 부하 높을 때 추가 부하. 복제본·정족수가 커질수록 느린 복제본을 만날 확률 증가(보통 7중 4, 9중 5 이하). 대규모 단절 시 정족수 구성 불가 → 느슨한 정족수(sloppy quorum, Cassandra의 consistency level ANY)로 아무 복제본이나 쓰기 허용.
리더리스 멀티리전

코디네이터 기반 멀티리전

Cassandra·ScyllaDB: 클라이언트가 로컬 리전 코디네이터 노드에 쓰면, 코디네이터가 자기 리전 모든 복제본 + 타 리전 복제본 1개에 전달(그게 나머지로 전파) → 리전 간 요청 중복 회피.

일관성 수준 선택

전 리전 정족수 / 리전별 정족수 / 로컬 리전 정족수. 로컬 정족수는 느린 타 리전 대기를 피하지만 stale 결과 가능성↑.

Riak: 클라이언트-노드 통신을 한 리전에 국한(n = 리전 내 복제본 수), 리전 간 복제는 다중 리더 스타일로 비동기.

동시 쓰기 감지

"happens-before" 관계

두 연산이 동시인지 어떻게 판단하나? A가 B보다 먼저 일어났다(happens-before) = B가 A를 알거나, A에 의존하거나, A 위에 쌓인 것. 어느 쪽도 상대보다 먼저가 아니면 동시(concurrent)이다.

A → B

나중 것이 먼저 것을 덮어쓴다.

B → A

역시 나중 것이 승리.

A ∥ B

동시 → 충돌, 해결 필요.

시간·상대성: 물리적으로 겹치는지는 중요치 않다. 네트워크가 느리거나 끊겨 서로 모른 채 일어났다면 — 빛의 속도로는 영향을 줄 수 있었더라도 — 두 연산은 동시다. (Lamport의 고전 논문 참조)
버전 벡터 · 장바구니 예제

인과 의존성을 포착하기

단일 복제본 알고리즘
  • 서버가 키마다 버전 번호를 두고 쓰기마다 증가.
  • 읽으면 덮이지 않은 모든 형제 + 최신 버전 반환. 쓰기 전 반드시 읽기.
  • 쓸 때 직전 읽기의 버전을 포함하고, 받은 값들을 병합해서 보낸다.
  • 서버는 그 버전 이하 값은 덮고, 더 높은 버전 값은 형제로 보존.

우유→달걀→밀가루→햄→베이컨을 두 클라이언트가 번갈아 추가해도, 옛 버전은 결국 덮이고 어떤 쓰기도 유실되지 않는다.

버전 벡터 (Version Vector)

복제본이 여럿이면 키마다 + 복제본마다 버전 번호가 필요. 각 복제본이 자기 번호를 올리고 다른 복제본들에서 본 번호도 추적 — 이 묶음이 버전 벡터다. (Riak 2.0의 dotted version vector = "causal context")

버전 벡터 덕분에 한 복제본에서 읽고 다른 복제본에 써도 안전하다. 형제가 생길 수 있으나 올바르게 병합하면 데이터는 잃지 않는다. ※ 버전 벡터 ≠ 벡터 클록 (미묘하게 다름; 복제본 상태 비교엔 버전 벡터가 정답)
6장 요약

복제 — 한눈에

세 가지 접근
  • 단일 리더 — 한 리더가 쓰기를 받아 스트림 전파. 이해 쉽고 강한 일관성. 팔로워 읽기는 stale 가능.
  • 다중 리더 — 여러 리더가 쓰기 수용, 서로 스트림 교환. 충돌 해결 필요, 일관성 약함.
  • 리더리스 — 여러 노드에 병렬로 쓰고 읽어 stale 노드를 감지·정정.
동기/비동기는 장애 시 동작에 큰 영향. 비동기 팔로워를 리더로 승격하면 최근 확정 데이터가 유실될 수 있다.
지연 일관성 모델
  • 읽기-후-쓰기 — 자기가 제출한 데이터는 항상 보인다.
  • 단조 읽기 — 본 적 있는 데이터보다 과거로 되돌아가지 않는다.
  • 일관된 접두 읽기 — 인과적으로 말이 되는 순서로 본다(질문→답).
다중 리더·리더리스는 버전 벡터로 동시 쓰기를 감지하고 CRDT 등 충돌 해결로 수렴시킨다. LWW·수동 해결도 가능. 다음 장: 샤딩 — 각 머신이 데이터의 일부만 저장.
