분산 캐싱의 한계점

⚠️ 일관성 유지의 어려움
여러 노드에서 데이터를 일관되게 유지하는 것은 어려운 문제. 지연시간-일관성 간 타협 필요
⚠️ 캐시 무효화의 복잡성
데이터 변경 시 캐시 제거/업데이트 과정이 복잡. 모든 노드가 변경을 인지하고 갱신해야 함
⚠️ 설정과 관리의 복잡성
노드 간 조율, 데이터 분산 전략 등 단일 노드 대비 운영 난이도 상승
⚠️ 네트워크 부하 문제
노드 간 업데이트 주고받는 과정에서 네트워크 부하 발생 (특히 지리적 분산 시)
⚠️ 캐시 데이터의 오래됨
동기화가 원활하지 않으면 업데이트 전 데이터를 반환하는 문제 발생
⚠️ 데이터 분배의 어려움
효과적인 분할 전략이 없으면 일부 노드 과부하, 일부 노드 유휴 상태
💸 높은 메모리 사용량 & 비용
대량 데이터 저장 시 메모리 사용량 ↑. 인프라 및 유지 관리 비용 고려 필요
✍️ 쓰기 작업 한계 & 접근 패턴
읽기 위주 환경에 적합. 쓰기가 빈번하면 일관성 유지 어려움. 캐시 히트율 낮으면 효과 저하

💡 결론: 도입 전 구체적인 요구사항을 살펴보고, 분산 캐싱이 정말 적절한지 검토 필요

6.2 분산 캐시 설계

요구사항을 기능적/비기능적으로 분류

기능적 요구사항

put(key, value) → 캐시에 추가
get(key) → 키로 값 조회

비기능적 요구사항

  • 높은 성능 — 낮은 지연시간, 높은 처리량
  • 높은 확장성 — 데이터/요청 증가에도 성능 유지. 노드 추가로 대응
  • 높은 가용성 — 장애 발생 시에도 서비스 중단 없이 안정 작동

캐시 삭제 정책

캐시 용량에 도달하거나 특정 조건 충족 시 어떤 데이터를 삭제할 지 결정하는 전략

삽입 순서 기반

FIFO (선입선출)
먼저 들어온 데이터를 먼저 삭제
LIFO (후입선출)
나중에 들어온 데이터를 먼저 삭제

접근 기반

MRU (최근 사용 항목 제거)
최근 사용 항목이 당분간 불필요할 것으로 판단되는 경우 효과적
⭐ LRU (가장 오래된 항목 제거)
가장 일반적 정책. 최근 사용 데이터를 다시 사용할 가능성이 높은 경우 효과적
LFU (최소 사용 빈도 제거)
접근 빈도 높은 데이터가 계속 빈번할 때 유리. 약점: 과거에 자주 사용했지만 지금은 거의 사용 안 되는 경우

LRU 캐시 설계

캐시 항목 수(N)가 정해져 있고, 추가/삭제 모두 O(1) 에 처리되어야 한다.

→ 이중 연결 리스트 + 해시맵 조합

10 / 15

LRU 읽기 흐름 — 3가지 케이스

Case 1: 캐시 미스 + 여유 있음
키가 해시맵에 없고 항목 수 < N

1. DB에서 valueObject 조회
2. 노드 생성 → 리스트 맨 앞 추가
3. 해시맵에 키 추가 (값 = 노드 포인터)
Case 2: 캐시 미스 + 가득 참
키가 해시맵에 없고 캐시 Full

1. 리스트 끝(Tail)의 가장 오래된 항목 확인
2. 리스트에서 삭제 + 해시맵에서 제거
3. Case 1과 동일 단계 수행
Case 3: 캐시 히트
키가 해시맵에 존재

1. 해시맵에서 키 검색 → 포인터로 노드 이동
2. 해당 노드를 리스트 맨 앞(Head)으로 이동

💡 쓰기는 DB에 직접 저장. 읽기에서만 캐시 미스 시 새 데이터 추가

valueObject : { "name": "John Doe", "city": "San Jose", "state": "California" }

시스템 통합: 배포 방식

방법 1: 서버 통합형

캐시를 애플리케이션 서버와 같은 머신에 함께 배치

장점

동일 머신 내 프로세스 간 호출로 빠른 조회

한계

머신 다운 시 해당 서버의 캐시 데이터 전체 손실

방법 2: Standalone형

캐시를 독립 서버 클러스터에 배포

장점

애플리케이션 서버와 독립적으로 확장 가능

한계

데이터 분배를 위해 로드밸런서, 해싱 전략 등이 필요

12 / 15

배포 방식 비교 & 요구사항 점검

구분서버 통합형Standalone형
성능네트워크 오버헤드 없이 직접 접근높은 처리량/확장성에 유리
확장성앱+캐시 함께 확장, 요구가 지나치게 높지 않을 때캐시 독립 확장 가능, 더 높은 확장성
자원 공유메모리/CPU 효율적 활용자원 충돌 없음, 대규모에 적합
유연성설정 간단, 기술 선택 제한적솔루션 자유 선택, 기술적 독립
운영 복잡성배포/관리 간단, 세부 요구 충족 어려울 수 있음복잡하나 앱 서버 무영향으로 수정 가능
비용자원 공유로 인프라 비용 ↓추가 인프라 비용 발생
네트워크 지연지연 최소화가 최우선일 때약간의 네트워크 지연 허용 가능할 때

비기능 요구사항 점검:

✅ 높은 성능 — 모든 연산 O(1)
✅ 높은 확장성 — 서버 수 증가 + 샤딩으로 대응
⚠️ 높은 가용성 — 서버 다운 시 데이터 손실 위험 → 데이터 복제로 해결

데이터 복제 활용 시 일관성 문제 발생 가능 → 가용성 vs 일관성 사이의 선택(trade-off)

대표 솔루션: Redis vs Memcached

구분RedisMemcached
특징다양한 데이터 구조 지원, 영구 저장 옵션, Pub/Sub, 트랜잭션, Lua 스크립팅단순하고 직관적 키-값 저장소, 가볍고 사용 쉬운 설계
활용 사례웹 캐싱, 실시간 분석, 세션 저장소, 리더보드/카운팅웹 캐싱, 세션 저장소, DB 결과 캐싱, 단순 키-값 분산
확장성샤딩 방식으로 수평 확장일관된 해싱으로 노드 추가를 통한 수평 확장
데이터 보존영구 저장 옵션 있음메모리 기반, 영구 저장 불가
데이터 구조String, List, Set, Hash, Sorted Set 등 다양단순 키-값만
사용 편의성다양한 기능만큼 학습 곡선 존재단순·직관적, 쉬운 통합/운영
노드 관계각 노드가 서로 상태를 인지. 복제, Master-Slave 구조, 장애 시 failover 지원각 노드가 완전 독립적, 서로 인지 못함. 데이터 복제 불가

핵심 정리

캐싱 — 자주 사용하는 데이터를 빠른 위치에 임시 저장하여 성능 개선
분산 캐싱 — 여러 노드에 분산 저장. 확장성·가용성 향상, 대규모 시스템에 적합
설계 핵심 — LRU(이중 연결 리스트 + 해시맵), O(1) 연산, 삭제 정책
배포 — 서버 통합형 vs Standalone형. 요구사항에 따라 선택
가용성 — 데이터 복제로 보장. 가용성 vs 일관성 trade-off 고려
Redis vs Memcached — 복잡한 기능 = Redis, 단순·빠른 캐싱 = Memcached
