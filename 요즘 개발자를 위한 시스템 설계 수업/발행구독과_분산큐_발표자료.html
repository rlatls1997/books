1. 분산 시스템의 발전 과정

방대한 데이터를 효과적으로 처리하고 배포하려면 안정적이고 확장성이 뛰어난 시스템이 필요하다.

💡 분산 큐는 여러 서버에서 분산된 큐로 메시지를 저장하고 전달하는 역할.
메시지를 보낸 순서대로 처리하거나, 각 메시지가 하나의 소비자만 대상으로 할 때 주로 사용.

1 / 19

분산 큐 설계 — 큐의 이점

분산 큐는 시스템 내 다양한 구성 요소가 메시지를 안정적으로 처리하고 전달할 수 있게 한다.

⚖️
부하 관리
생성자와 소비자 사이의 완충 역할. 메시지 폭증 상황에서도 시스템 과부하 방지
🔗
낮은 결합도
각 부분이 독립적으로 동작. 생성자와 소비자가 서로의 상태를 몰라도 되므로 유지보수 용이
🛡️
신뢰성과 일관성
소비자가 처리에 실패하더라도 큐가 메시지를 다시 전달하여 데이터 일관성 유지
2 / 19

분산 큐 설계 시 고려사항

📈
확장성
작업량이나 메시지 크기가 많아도 안정적으로 감당할 수 있어야 함
🔄
장애 허용성
시스템 일부에 장애가 발생해도 큐가 지속적으로 작동해야 함
✅
메시지 일관성
메시지가 한 번만 정확히 전달되고 올바른 순서로 처리되도록 보장
3 / 19

분산 큐의 주요 특징

🎛️
큐 관리자
메시지 분배를 책임지는 핵심 구성 요소. 메시지 순서 유지, 적절한 소비자에게 전달, 실패 시 재전송 관리
💾
메시지 저장소
소비자가 처리하기 전까지 메시지를 보관하는 공간. 높은 신뢰성과 빠른 읽기/쓰기 필요
⚖️
로드 밸런싱
특정 소비자에게 메시지가 몰리는 것을 방지. 각 소비자의 작업량과 처리 능력 고려하여 균형 분배
※ Kafka는 파티션↔소비자 고정, RabbitMQ는 여러 컨슈머가 하나의 큐를 처리하여 분배 가능 (단 순서 보장 제한)
🔧
장애 허용과 복구
실패한 메시지나 처리 오류를 감지하고, 재전달 또는 다른 경로로 전송하여 안정성 확보
🚀
확장성
메시지와 소비자가 늘어나더라도 성능 저하 없이 효과적으로 처리하며 확장 가능
4 / 19

아키텍처 설계 시 고려 사항

📋 메시지 순서 보장
요구사항에 따라 최소 한번 또는 정확히 한번 전달 설정. 데이터 처리와 우선순위, 구성요소 간 신뢰성에 영향
💿 데이터 영속성
메시지를 얼마나 오래 저장할지 결정. 소비자 확인 시까지 보관 vs 일정 수준의 손실 허용. 데이터 무결성과 성능 간 균형
🔒 보안
메시지 무결성 보호, 큐 접근 사용자 제어를 통한 개인정보보호 요건 준수
🧩 기술 스택 선택
언어 지원, 성능 요구 사항, 기존 인프라와의 호환성을 고려하여 선택
📝 메시지 형식 설계
메시지 크기, 직렬화/역직렬화, 디버깅용 메타데이터 구성. 큐의 성능과 사용 편의성에 영향
🔄 장애 허용성
재시도 메커니즘, 데드 레터 큐(DLQ) 같은 전략으로 지속 운영 및 데이터 손실 최소화

💡 분산 큐 설계는 기술적 요소와 실질적 요구 사항 간의 균형을 고려해야 한다.

2. 발행/구독 시스템 설계

생성자가 특정 토픽에 메시지를 발행하면, 해당 토픽을 구독한 소비자가 메시지를 받아보는 구조.

💡 토픽 = 소비자가 메시지를 받을 수 있도록 설정된 카테고리/채널

하나의 토픽에 다수의 생성자와 다수의 소비자가 존재할 수 있다.

6 / 19

Pub/Sub 주요 특징

🔗
낮은 결합도
생성자와 소비자가 느슨하게 연결. 생성자가 소비자를 알 필요 없음 → 유연성·확장성 향상
🚀
확장성과 효율성
대량의 메시지를 여러 소비자에게 효율적으로 분배. 대규모 분산 시스템에 적합
🔀
메시지 처리의 유연성
소비자가 다양한 방식으로 처리 가능. 실시간 분석부터 이벤트 기반 아키텍처까지 폭넓은 활용
7 / 19

Pub/Sub 아키텍처 설계 요소

📁
토픽 관리
토픽 생성·관리·삭제 방식을 명확히 정의. 시스템 요구사항 변화에 따라 토픽을 동적으로 생성/삭제하는 방안 고려
🔀
메시지 라우팅
메시지를 적절한 소비자에게 전달할 수 있는 라우팅 메커니즘 개발
👥
소비자 관리
소비자 등록, 어떤 토픽을 구독할지 관리
📊
서비스 품질 (QoS)
메시지 보장 수준 결정: 최대 한 번 / 최소 한 번 / 정확히 한 번. 메시지 순서 관리 방법도 고려
⚖️
확장성과 로드 밸런싱
생성자·소비자가 늘어나더라도 원활하게 처리할 수 있는 확장성 확보
8 / 19

설계 핵심 — 확장성

작업량이 증가해도 성능 저하 없이 처리할 수 있는 능력

↔️
수평 확장
시스템에 장비를 추가하는 방식. 유연성과 안정성이 뛰어나고 서버 추가로 용량을 쉽게 확장
↕️
수직 확장
기존 장비에 CPU, 메모리 추가. 단기적으로 간단하나 수평확장만큼 유연하거나 안정적이지 않음
⚖️
동적 로드 밸런싱
메시지 브로커가 여러 구독자 사이에 부하를 골고루 분배. 발행자/구독자는 분배를 신경쓰지 않아도 됨
🗂️
토픽 파티셔닝
하나의 토픽을 여러 파티션으로 분할하여 메시지 부하를 시스템 전체에 고르게 분산 → 처리 성능·처리량 향상
9 / 19

설계 핵심 — 신뢰성

메시지 전달과 시스템 복원력을 유지하는 데 필수

✉️
메시지 전달 보장
최대 한 번 / 최소 한 번 / 정확히 한 번 전달 설정. 데이터 손실·중복 처리 방식에 영향
🔄
장애 허용성
데이터를 여러 노드에 복제 저장, 재시도 메커니즘 설계로 장애 시에도 원활한 동작
🔢
메시지 순서
순서가 중요한 애플리케이션에서 정확한 순서 보장은 신뢰성 확보의 필수 요소

소비자 관리와 메시지 라우팅

👤 소비자 등록 및 관리
특정 토픽 구독/취소, 구독 상태 관리 지원
🔀 효율적인 메시지 라우팅
적절한 소비자에게 전달하면서 지연 최소화, 자원 효율적 사용 알고리즘 설계

💡 서비스 품질(QoS): 시스템 특성에 따라 정확한 전달 vs 낮은 지연시간 중 우선순위가 다를 수 있음

3. 카프카 (Kafka)

대용량 실시간 데이터를 내구성, 속도, 확장성을 고려하여 처리하기 위해 설계된 분산 스트리밍 플랫폼

📌
토픽 (Topic)
레코드(메시지)가 분류되는 카테고리. 생성자와 소비자는 토픽을 통해 메시지 송수신
↔️
생성자 & 소비자
생성자는 토픽으로 메시지를 보내고, 소비자는 토픽에서 읽어 처리. 독립적 확장 가능
🖥️
브로커 (Broker)
카프카 클러스터를 구성하는 서버. 데이터 저장, 소비자 관리, 오프셋 추적
🗂️
파티션 & 복제
토픽을 여러 파티션으로 분할, 데이터를 복제하여 여러 브로커에 분산 → 장애 대비 + 처리량 향상

🐘 주키퍼(ZooKeeper): 파티션 리더 브로커 결정, 브로커 상태 추적
Kafka 2.8+부터 KRaft 프로토콜로 내부 관리 가능 (주키퍼 제거 방향)

카프카 배포 단계

1️⃣ 클러스터 설정
클러스터 설정 — 브로커 여러 개 구성하여 장애 허용성·고가용성 확보. 브로커 수 결정, 부하 분산 세부 설정
주키퍼 통합 — 클러스터 정보 관리, 브로커 간 역할 분담
2️⃣ 토픽 생성 및 설정
토픽 생성 — 파티션 수와 복제 계수를 적절히 설정
토픽 관리 — 설정 수정/삭제, 성능 모니터링 방법 구비
3️⃣ 생성자와 소비자 구현
생성자 — 카프카 토픽으로 메시지를 보내기 위한 구현
소비자 — 토픽 구독하여 처리. 동시성 처리와 오프셋 관리에 중점
4️⃣ 모니터링과 유지 보수
클러스터 모니터링 — 상태·성능 지표 확인 도구 활용
성능 튜닝 — 생성자/소비자 설정, 네트워크, 디스크 I/O 최적화

4. 카프카 스트림

카프카 위에서 데이터를 실시간으로 처리하고 분석하는 애플리케이션을 만들기 위한 도구

🔌
카프카와 완전한 통합
토픽에서 데이터를 자연스럽게 읽고 쓰기 가능. 파티션·복제 기능 활용으로 장애 허용성·확장성 향상
🧩
고수준 DSL
스트림 처리 애플리케이션을 쉽게 개발할 수 있는 다양한 연산 지원
⚙️
저수준 Processor API
복잡한 데이터 처리나 처리 구조를 세밀하게 제어할 때 사용
📦
상태 기반 처리
시간이 지나도 애플리케이션 상태를 유지하고 업데이트
✅
정확히 한 번 처리
카프카의 트랜잭션 기능을 활용하여 중복 처리·누락 없이 정확한 처리 보장
📈
확장성 및 장애 허용성
여러 인스턴스를 추가하여 수평 확장 가능
14 / 19

스트림 처리 토폴로지

스트림 처리 애플리케이션은 토폴로지라는 형태로 정의된다.

토폴로지 = 스트림 프로세서(노드) + 스트림(엣지)으로 구성된 그래프

각 프로세서는 입력 데이터를 처리하여 출력 스트림을 생성하며, 프로세서 간 연결 관계를 정의한다.

카프카 스트림 활용 사례

📊
실시간 데이터 분석
스트리밍 데이터를 실시간으로 분석·집계
⚡
이벤트 기반 아키텍처
이벤트 발생 시 실시간으로 반응하는 시스템 구축
🔄
데이터 보강 및 변환
스트림 데이터를 변환하고 외부 데이터로 보강
🚨
부정행위 감지
실시간으로 부정행위를 식별하고 방지
🔃
스트리밍 ETL
스트리밍 데이터를 실시간 Extract → Transform → Load하여 지속적 데이터 통합·처리
✨
핵심 장점
효과적이고 유연한 스트림 처리 애플리케이션 구축 가능

5. 키네시스 (Kinesis)

AWS가 제공하는 클라우드 기반 실시간 데이터 스트리밍 서비스

💡 Kafka = 직접 클러스터 구축·관리 (브로커, 파티션, 레플리카, 디스크 등)
Kinesis = AWS 완전관리형 서비스 — 인프라 운영 없이 샤드 수만 설정

같은 문제를 해결하지만 "직접 운영" vs "매니지드"라는 축에서 반대편에 있음

17 / 19

Kafka vs Kinesis

구분KafkaKinesis
용량 확장브로커 추가, 파티션 재분배 등 직접 수행샤드 수 조정 또는 On-Demand 모드로 자동 스케일링
데이터 보존보존 기간 자유 설정 가능, 무제한도 가능기본 24시간, 최대 365일 (비용 증가)
생태계Kafka Streams, Kafka Connect, Schema Registry 등 풍부한 생태계AWS 생태계 (Lambda, Firehose, Analytics)와의 강력한 통합
처리량파티션당 처리량 제한이 사실상 하드웨어 한계까지샤드당 1MB/s 입력, 2MB/s 출력 제한
비용 구조인프라 비용 + 운영 인력 필요샤드 시간 + PUT 요청 수 기반 종량제
적합 상황Kafka 운영 역량 보유, 세밀한 제어 필요 시AWS 환경에서 운영 부담 없이 빠르게 구축 시

핵심 정리
