# 7장. 발행/구독과 분산 큐
방대한 데이터를 효과적으로 처리하고 배포하려면 안정적이고 확장성이 뛰어난 시스템이 필요하다. 

이 과정에서 분산 시스템이 중요한 역할을 하고, 특히 발행/구독 시스템과 분산 큐 등 메시징 패턴을 핵심적으로 활용한다.

이번 장에서는 다음 내용을 다룬다.
- 분산 시스템의 발전 과정
- 분산 큐 설계
- 발행/구독 시스템 설계
- 카프카(Kafka)
- 키네시스(Kinesis)


## 1. 분산 시스템의 발전 과정
분산 시스템은 모든 구성 요소가 하나의 서비스 안에 밀접하게 연결된 모놀리식 아키텍처에서  
각 서비스가 독립적으로 분리된 상태로 메시지를 주고 받으며 소통하는 마이크로 서비스 아키텍처로 발전해왔다.

분산 큐는 여러 서버에서 분산된 큐로 메시지를 저장하고 전달하는 역할을 함

이런 큐는 메시지를 보낸 순서대로 처리해야 하거나, 
각 메시지가 하나의 소비자만 대상으로 할 때 주로 사용한다.

다음 절에서 분산 큐와 발행/구독 시스템을 설계하고 구현하는 방법을 살펴본다


### 분산 큐 설계
분산 큐는 시스템 내 다양한 구성 요소가 메시지를 안정적으로 처리하고 전달할 수 있게 한다.

부하를 고르게 분산하고, 수요가 증가해도 시스템이 유연하게 확장될 수 있다.

분산 큐를 활용할 때 얻을 수 있는 주요 이점을 살펴보자

#### 큐를 활용하는 이점
- 부하 관리 : 큐는 메시지의 생성자와 소비자 사이에서 완충 역할을 하며 메시지가 폭증하는 상황에서도 시스템이 과부하에 걸리지 않게 도와준다.
- 구성 요소 간 결합도 낮추기 : 큐를 사용하면 시스템의 각 부분이 서로 독립적으로 동작할 수 있다. 메시지의 생성자와 소비자가 서로의 상태를 신경쓰지 않아도 되므로 시스템을 유지 보수하기 더 쉬워진다.
- 신뢰성과 일관성 : 메시지가 안전하게 처리되도록 할 수 있다. 소비자가 메시지 처리에 실패하더라도 큐가 이를 다시 전달해서 데이터 일관성을 유지할 수 있다.

다음으로 큐 시스템을 설계할 때 어떤점을 고려해야 할지 살펴본다

#### 분산 큐 설계 및 구현하기
분산 큐를 설계할때는 부하 관리, 안정성, 메시지를 전달하는 바식을 어떻게 구현할지 고민해야 한다.

- 확장성 : 큐가 처리해야할 작업량이나 메시지 크기가 많아도 이를 안정적으로 감당할 수 있어야 한다.
- 장애 허용성 : 시스템 일부에 장애가 발생하더라도 큐가 지속적으로 작동할 수 있어야 한다.
- 메시지 일관성 : 메시지가 한 번만 정확히 잔달되고 올바른 순서로 처리되도록 해야한다.

다음으로 분산에 어떤 특징이 있는지 알아본다.

#### 분산 큐의 주요 특징
분산 큐는 메시지 흐름을 효율적으로 관리하려고 각 구성 요소를 유기적으로 작동하도록 설계했다

- 큐 관리자 : 큐에서 메시지 분배를 책임지는 핵심 구성 요소이다. 메시지 순서를 유지하고, 적절한 소비자에게 전달하며, 처리에 실패했을 때 재전송을 관리하는 역할을 한다.
- 메시지 저장소 : 소비자가 메시지를 처리하기 전까지 보관되는 공간. 높은 신뢰성을 바탕으로 대량의 데이터를 처리할 수 있어야 하며 빠른 읽기 및 쓰기 작업으로 메시지 처리를 효율적으로 지원해야 한다.
- 로드 밸런싱 : 분산 큐는 특정 소비자에게 메시지가 몰리는 일을 방지하기 위해 부하를 고르게 분산하는 것이 중요하다. 각 소비자의 현재 작업량과 처리 능력을 고려해서 메시지를 균형 있게 분배합니다.
  (키를 기반으로 메시지가 특정 파티션으로 들어가고, 파티션은 이미 정해져있는 소비자가 처리하는데 소비자에게 메시지가 고르게 분산되는게 가능한가? p.216)  
kafka 기준으로는 옳지 않은 설명일 수 있으나, RabbitMQ같은 경우는 파티션이라는 단위가 없고, 여러 컨슈머가 하나의 큐를 처리하기 때문에 분배가 가능함. 하지만 단일 컨슈머가 아니라면 컨슈머의 기준으로 순서 보장이 안됨) 
- 장애 허용과 복구 : 분산 큐는 시스템에 문제가 생겨도 안정적으로 작동해야 함. 실패한 메시지나 처리 오류를 감지하고, 메시지를 다시 전달하거나 다른 경로로 보낼 수 있는 기능으로 안정성과 신뢰성을 확보.
- 확장성 : 메시지와 소비자가 늘어나더라도 성능 저하 없이 이를 효과적으로 처리하며 확장 가능해야 한다.

다음으로 분산 큐 시스템의 각 구성요소가 어떻게 연계되고 작도하는지 살펴본다.

#### 아키텍처를 설계할 때 고려할 사항

<img src="https://github.com/user-attachments/assets/2f06cf2d-45de-4b74-b396-242b180c0ac3" width="600">

분산 큐 시스템을 설계할 때는 여러 아키텍처 요소를 꼼꼼히 살펴야 한다. 
이런 요소들은 시스템을 효율적이고 확장 가능하며 안정적으로 운영하는데 필수이다.

- 메시지 순서 보장 : 분산 큐 시스템 설계에서 중요한 요소 중 하나는 메시지의 순서와 전달 방식을 설정하는 것이다. 요구사항에 따라 메시지를 최소 한번 또는 정확히 한번 메시지를 전달해야 할 수도있다. 이는 시스템이 데이터를 처리하고 우선순위를 정하는 방식에 영향을 미치며, 구성요소간 신뢰성 있고 정확한 통신을 유지하는 데 핵심역할을 한다.
- 데이터영속성 : 메시지를 시스템에 얼마나 오래 저장할지 결정하는 것도 중요하다. 애플리케이션에 따라 소비자가 메시지를 확인할때까지 보관해야 하는 경우도 있지만, 성능을 우선시하려고 일정 수준의 메시지 손실을 허용하는 방식이 필요할 수도 있다. 데이터 무결성과 시스템 성능 간 균형을 신중히 고려해야한다.
- 보안 : 메시지 무결성을 보호하고 안전한 환경을 유지하려면 보안 조치를 마련해야 한다. 큐에 접근할 수 있는 사용자를 제어하여 개인정보보호 요건을 준수할 수 있다.
- 기술 스택 선택 : 어떤 기술스택을 사용할 지 결정할 수 있다. 언어 지원, 성능 요구 사항, 기존 인프라와의 호환성 같은 요소에 따라 다르다.
- 메시지 형식 설계 : 메시지의 크기를 정하고 데이를 직렬화/역직렬화 하거나 디버깅에 필요한 메타데이터를 구성하는 작업. 큐의 성능과 사용 편의성에 영향을 준다
- 장애 허용성 : 구성 요소에 장애가 발생하면 이를 처리하기 위한 재시도 메커니즘이나 처리 불가능한 메시지를 관리하는 데드 레터 큐 같은 전략을 도입해야한다. 이것으로 시스템을 지속적으로 운영하고 데이터 손실을 최소화할 수 있다.

결론적으로 
분산 큐를 설계하는 과정은 기술적 요소와 실질적 요구 사항 간에 균형을 고려해야한다.  


## 2. 발행/구독 시스템 설계
발행/구독 모델은 메시지 전송 방식 중 하나로,  
생성자가 특정 토픽에 메시지를 발행하면 소비자가 해당 토픽에 구독한 메시지를 받아 볼 수 있는 구조이다.

토픽은 소비자가 메시지를 받을 수 있도록 설정된 카테고리나 채널의 개념이다.

발행/구독 시스템의 주요 특징을 알아보자


### 발행/구독 시스템의 주요 특징
발행/구독 시스템은 생성자와 소비자를 독립적으로 확장할 수 있도록 설계되며,  
확장성과 신뢰성을 보장해준다.

특징은
- 생성자와 소비자 간의 낮은 결합도 : 생성자와 소비자가 느슨하게 연결되어 있다. 생성자가 소비자를 알 필요가 없기 때문에 시스템의 유연성과 확장성이 향상된다.
- 확장성과 효율성 : 발행/구독 모델은 대량의 메시지를 처리하고 이를 여러 소비자에게 효율적으로 분배할 수 있어 대규모 분산 시스템에 적합하다.
- 메시지 처리의 유연성 : 소비자는 메시지를 다양한 방식으로 처리할 수 있어서 실시간 데이터 분석부터 이벤트 기반 아키텍처까지 다양한 활용이 가능하다.

발행/구독 시스템은 기본적으로 확장성과 신뢰성, 독립적인 구조를 갖추어야 한다.  
이를 위해 토픽 관리, 메시지 라우팅, 소비자 관리, 서비스 품질 정의 등 중요한 요소로 어떻게 아키텍처를 설계할지 신중히 결정해야 한다.
- 토픽 관리 : 토픽을 생성-관리-삭제하는 방식을 명확히 정리해야한다. 메시지를 보낼 목적지로, 그리고 메시지를 받을 채널로 토픽을 사용한다. 시스템과 사용자 요구 사항이 바뀌면서 토픽을 동적으로 생성하거나 삭제하는 방안도 생각해야한다.
- 메시지 라우팅 : 메시지를 적절한 소비자에게 전달할 수 있는 라우팅 메커니즘을 개발하는 것이 중요하다.
- 소비자 관리 : 소비자를 등록하고 각 소비자가 어떤 토픽을 구독할지 관리하는 것도 중요하다.
- 서비스 품질 기술 : 메시지를 어느 수준까지 보장할지 결정하는 것이 중요하다. 최대 한 번만 보낼지, 최소 한 번 이상 보장할지, 정확히 한번만 전달되도록 할지 설정할 수 있다 그리고 메시지 순서를 어떻게 관리할지 고려해야 시스템의 신뢰성과 효율성을 높일 수 있다.
- 확장성과 로드 밸런싱 : 생성자와 소비자가 늘어나더라도 이를 원활하게 처리할 수 있도록 확장성을 갖추어야 한다.

하나의 토픽에 다수의 생성자와 다수의 소비자가 존재할 수 있다.


### 발행/구독 시스템을 설계할 때 고려할 사항
발행/구독 시스템을 설계할 때 특히 신경써야 두 가지 핵심 요소는 확장성과 신뢰성이다. 
시스템이 방대한 양의 메시지를 안정적으로 처리하면서, 동시에 운영환경이 변하더라도 안정적인 성능을 유지하는게 조건이다.

`확장성`이란 작업량이 증가하더라도 시스템이 성능 저하 없이 자연스럽게 처리할 수 있는 능력을 의미한다.  
확장성을 결정 짓는 요소를 하나씩 살펴보자.

- 수평 확장과 수직 확장
  - 수평 확장 : 시스템에 장비를 추가하는 방식으로 유연성고 안정성이 뛰어나다. 필요한 만큼 서버를 추가하여 시스템 용량을 쉽게 늘릴 수 있다.
  - 수직 확장 : 기존 장비에 cpu, 메모리 같은 자원을 추가하는 것이다. 단기적으로는 간단하나 수평확장만큼 유연하거나 안정적이지 않다.
- 동적 로드 밸런싱 : 발행/구독 모델에서는 주로 메시지 브로커가 로드밸런싱을 수행하여 여러 구독자 사이에 메시지 부하를 골고루 나눈다. 브로커덕분에 발행자와 구독자는 메시지분배나 부하를 신경쓰지 않아도 된다.
- 토픽 파티셔닝 : 하나의 토픽을 작고 관리하기 쉬운 여러 파티션으로 나눌 수 있다. 토픽을 파티션으로 나누면 메시지 부하를 시스템 전체에 고르게 분산할 수 있어서 처리 성능이 좋아지고 처리량도 올라간다.

`신뢰성`을 확보하는 것은 메시지 전달과 시스템 복원력을 유지하는 데 필수이다. 
신뢰성을 위해 고려해야할 요소를 살펴보자
- 메시지 전달 보장 : 메시지 무결성을 위해 다양한 수준의 전달 보장을 구현하는 것이 중요하다. 최대한번, 최소한번, 정확히 한번 전달되도록 설정할 수 있다. 이런 설정은 데이터 손실이나 중복을 어떻게 처리할지에 영향을 미친다.
- 장애 허용성 : 장애 상황에서도 원활히 동작하기 위해 데이터는 여러 노드에 복제해서 저장하고, 메시지를 다시 전송할 수 있는 재시도 메커니즘을 설계해야 한다.
- 메시지 순서 : 메시지 순서가 중요한 애플리케이션에서 정확한 순서를 보장하는 것은 신뢰성 확보를 위해 필수이다.

#### 소비자 관리와 메시지 라우팅
발행/구독 시스템에서는 소비자를 효과적으로 관리하고 메시지를 올바르게 전달하는 것이 핵심이다. 
이를 위해 다음 상황을 고려해야 한다.

- 소비자 등록 및 관리 : 소비자가 특정 토픽을 구독하거나 구독을 취소하고 자신의 구독 상태를 관리할 수 있도록 지원
- 효율적인 메시지 라우팅 : 메시지를 적절한 소비자에게 전달하면서 지연 시간을 최소화하고 자원을 효율적으로 사용할 수 있는 알고리즘을 설계하는 것

#### 서비스 품질 수준
시스템의 특성과 요구 사항에 따라 필요한 서비스 품질 수준은 다를 수 있다.  

어떤 시스템은 메시지가 정확히 전달되는 것이 중요할 수 있고, 
어떤 시스템은 지연시간을 줄이는 데에 더 중점을 둘 수 있다.

## 3. 카프카
대용량, 실시간 데이터를 내구성, 속도, 확장성을 고려하여 처리하기 위해 설계된 분산 스트리밍 플랫폼이다.  

핵심 개념을 살펴보면 
- 토픽 : 카프카에서 레코드(메시지)는 토픽이라는 카테고리고 분류된다. 생성자와 소비자는 토픽을 통해 메시지를 보내고 받을 수 잇다.
- 생성자와 소비자 : 생성자는 특정 토픽을 메시지를 보내고, 소비자는 데이터를 토픽에서 읽어 처리한다. 생성자와 소비자는 독립적으로 확장가능할 수 있도록 설계되어 있다.
- 브로커 : 카프카 클러스터는 브로커라고 하는 하나 이상의 서버로 구성된다. 브로커는 데이터를 저장하고, 소비자를 관리하며, 데이터가 어디까지 소비되었는지 추적한다.
- 파티션과 복제 : 각 토픽은 여러 파티션으로 나눌 수 있고, 데이터를 복제하여 여러 브로커에 분산시켜 장애에 대비하고 처리량을 높을 수 있다.

카프카에서 주키퍼는 각 파티션의 리더 브로커를 결정하고, 브로커들의 상태를 추적하는 역할을 한다.  
(카프카 2.8 버전부터 클러스터의 메타데이터 관리와 브로커 조율을 카프카 컨트롤러 브로커와 카프카 래프트 합의 프로토콜 같은 내부 구성 요소가 담당한다.)

### 분산 시스템에서 카프카의 중요성
카프카는 대규모 데이터를 낮은 지연 시간으로 처리해야하는 현대 분산 시스템에서 핵심적인 역할을 한다.  
분산 환경에서 카프카를 배포하려면 설정 및 성능 최적화를 위한 여러 단계를 거챠야한다

1. 카프카 클러스터 설정하기
- 클러스터 설정 : 카프카 클러스터를 설정하려면 브로커를 여러 개 수정하여 장애 허용성과 높은 가용성을 확보해야 한다. 이를 위해 브로 개수를 정하고 부하를 고르게 분산할 수 있도록 각 브로커를 세부적으로 설정해야 한다.
- 주키퍼 통합 설정 : 클러스터 정보를 관리하고 브로커 간 역할을 분담하기 위해 주키퍼 설정

2. 토픽 생성 및 설정하기
- 토픽 생성 : 성능과 안정성을 높이기 위해 파티션 수와 복제 계수를 적절히 설정하여 토픽을 생성한다
- 토픽 관리 : 토픽 설정을 수정하거나 삭제하고 성능을 모니터링하기 위한 방법을 구비해야 한다.

3. 카프카 생성자와 소비자 구현하기
- 생성자 구현 : 카프카 토픽으로 메시지를 보내기 위해 생성자를 구현해야 한다.
- 소비자 구현 : 토픽을 구독하여 메시지를 처리하기 위해 소비자를 구현해야 한다. 동시성 처리와 오프셋 관리에 중점을 두고 메시지를 안정적으로 처리할 수 있게 구현해야 한다.

4. 모니터링과 유지 보수
- 클러스터 모니터링 : 시스템을 안정적으로 운영하기 위해 클러스터 상태와 성능 지표를 확인할 수 있는 도구를 활용하여 모니터링해야한다.
- 성능 튜닝 : 생성자와 소비자 설정, 네트워크, 디스크 입출력 속도를 최적화하는 등 성능을 개선하는 방법을 적용해야 한다.

다음으로 카프카 스트림을 알아본다.

## 4. 카프카 스트림
카프카 위에서 데이터를 처리하기 위해 설계된 도구.

테이터를 실시간으로 처리하고 분석하는 애플리케이션을 만들 수 있다.

주요 특징을 보면
- 카프카와 완전한 통합 : 카프카와 완전히 통합되어 카프카 토픽에서 데이터를 자연스럽게 읽고 쓰는 작업이 가능하다. 이 과정에서 카프카의 파티션 및 복제 기능을 활용하여 시스템의 장애 허용성과 확장성을 높일 수 있다.
- 고수준 DSL : 스트림 처리 애플리케이션을 쉽게 개발할 수 있도록 다양한 연산을 지원하는 고수준 DSL을 제공한다.
- 저수준 Processor API : 복잡한 데이터 처리나 처리 구조를 세밀하게 제어할 때 사용할 수 있는 저수준 Processor API도 제공한다.
- 상태 기반 처리 : 카프카 스트림은 시간이 지난 후에도 애플리케이션 상태를 유지하고 업데이트할 수 있도록 상태 기반 처리를 지원한다.
- 정확히 한 번 처리 : 데이터 처리의 신뢰성을 높이려고 레코드가 단 한번만 처리되도록 보장한다. 실패하더라도 중복 처리나 누락 없이 데이터를 정확히 처리하려고 카프카의 트랜잭션 기능을 활용한다.
- 확장성 및 장애 허용성 : 카프카 스트림 애플리케이션은 여러 인스턴스를 추가하여 수평 확장이 가능하다.

### 스트림 처리 토폴로지
카프카 스트림에서 스트림 처리 애플리케이션은 토폴로지라는 형태로 정의된다. 
토폴로지는 스트림 프로세서(노드) 와 스트림(에지)으로 구성된 그래프이다.

각 프로세서는 입력 데이터를 처리하여 출력 스트림을 생성하며, 
토폴로지는 데이터가 프로세서를 거치면서 흐르는 방식와 프로세서 간 연결 관계를 정의한다.

#### 카프카 스트림을 활용한 사례
카프카 스트림은 실시간 데이터 처리에 적합하며 다음 상황에서 활용할 수 있다.
- 실시간 데이터 분석 
- 이벤트 기반 아키텍처
- 데이터 보강 및 변환 : 
- 부정행위 감지 : 실시간으로 부정행위를 식별하고 방지할 때
- 스트리밍 ETL : 스트리밍 데이터를 실시간으로 추출(Extract), 변환(Transform), 적제(Load)하여 지속적인 데이터 통합과 처리가 가능하다.

카프카 스트림을 활용하면 효과적이고 유연함을 갖춘 스트림 처리 애플리케이션을 만들 수 있다.

## 5. 키네시스
키네시스는 AWS가 제공하는 클라우드 기반 서비스이다.

Kafka는 직접 클러스터를 구축하고 브로커 수, 파티션 수, 레플리카, 디스크 용량 등을 직접 설계하고 관리해야 한다. 
반면 Kinesis는 AWS 완전관리형 서비스라서 인프라 운영이 없고, 샤드 수만 설정하면 됩니다.

주요 차이점:
용량 확장 — Kafka는 브로커 추가, 파티션 재분배 등을 직접 해야 하는데, Kinesis는 샤드 수를 늘리거나 on-demand 모드를 쓰면 자동으로 스케일링됩니다.
데이터 보존 — Kafka는 보존 기간을 자유롭게 설정할 수 있고 무제한도 가능합니다. Kinesis는 기본 24시간이고 최대 365일까지 늘릴 수 있지만 비용이 올라갑니다.
생태계 — Kafka는 Kafka Streams, Kafka Connect, Schema Registry 등 풍부한 생태계가 있어요. Kinesis는 AWS 생태계(Lambda, Firehose, Analytics)와의 통합이 강점입니다.
처리량 — Kafka는 파티션당 처리량 제한이 사실상 하드웨어 한계까지인데, Kinesis는 샤드당 초당 1MB 입력 / 2MB 출력이라는 명확한 제한이 있습니다.
비용 구조 — Kafka는 인프라 비용 + 운영 인력이 필요하고, Kinesis는 샤드 시간 + PUT 요청 수 기반 종량제입니다.
정리하면, 이미 Kafka 운영 역량이 있고 세밀한 제어가 필요하면 Kafka가 낫고, AWS 환경에서 운영 부담 없이 빠르게 스트리밍 파이프라인을 구축하고 싶으면 Kinesis가 편합니다. 같은 문제를 해결하지만 "직접 운영 vs 매니지드"라는 축에서 반대편에 있는 거예요.
