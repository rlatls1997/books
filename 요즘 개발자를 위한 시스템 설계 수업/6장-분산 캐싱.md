# 6장. 분산 캐싱

애플리케이션이 복잡하고 사용자 수가 늘어나면서 데이터 처리에 병목 현상 발생 가능.  
이런 성능 문제를 해결하기 위한 방법 => `캐싱`

`캐싱`은 사용자와 가까운 위치에 데이터를 임시 저장하여 빠르게 접근할 수 있도록 하는 기술

애플리케이션이 커지면서 단일 캐시만으로는 충분하지 않을 수 있음.  
이때 `분산 캐싱`이 필요

이 장에서는  
캐싱 원리를 살펴보고 분산 캐싱이란 개념을 알아본다.  
캐싱의 중요성, 작동 방식, 활용 분야 등

## 6.1 캐싱 정의
캐싱 : 자주 사용하거나 연산 비용이 많이 드는 데이터를 더 빠르게 접근할 수 있는 위치에 저장하고 관리하는 기술. 
목적은, 데이터를 가져오는데에 필요한 시간, 리소스를 줄이는 것

#### 캐싱의 주요 개념
- 캐시 : 자주 접근하는 데이터, 리소스의 임지 저장 공간.
  - 하드웨어 캐시 (ex, CPU캐시)
  - 소프트웨어 캐시 (ex, 인 메모리 캐시)
- 캐싱된 데이터 : 캐시에 저장된 데이터
- 캐시 히트 : 요청한 데이터가 캐시에 있을 때를 의미 -> 원본 데이터 접근할 필요 없이 빠른 처리 가능
- 캐시 미스 : 캐시에 없을 때를 의미 -> 원본 데이터 조회 필요
- 캐시 제거 : 캐시가 가득 차면 새로운 데이터를 저장하기 위해 기존 데이터 제거

### 분산 캐싱 정의
분산 캐싱 : 여러 서버나 노드에 자주 사용하는 정보를 메모리에 분산 저장하는 기술.  
핵심 목표는 디스크 기반 저장 시스템에서 데이터를 읽는 데 걸리는 속도 문제를 해결하는 것


### 분산 캐싱과 일반 캐싱의 차이점
|       | 일반 캐싱                                               | 분산 캐싱                                                                            |
|-------|-----------------------------------------------------|----------------------------------------------------------------------------------|
| 적용 범위 | 단일 시스템 내 로컬 캐시에 저장                                  | 분산 캐싱은 캐싱 개념을 네트워크로 연결된 여로 노드나 서버로 확장한 방식.<br/> 캐시가 여러 기기에 분산되어 있고 노드간 데이터 공유 가능 |
| 아키텍처  | 데이터가 로컬 캐시에 저장됨. 동일한 기기 내에 있기 때문에 애플리케이션에서 바로 접근 가능 | 여러 캐시 노드로 구성된 네트워크를 기반으로 작동, 노드간 데이터 공유 가능, 수평 확장 가능                             |
| 확장성   | 소규모 환경에 적합.                                         | 대규모 시스템 성능 문제를 해결하기 위해 설계됨. 성능 및 확장성을 높이는 데 유리                                   |
| 활용 사례 | 애플리케이션 성능 개선 목적, 데이터베이스 접근 시간 줄이기 등                 | 여러 시스템이 데이터를 공유해야하는 환경에 효과적                                                      |
|일관성 및 동기화|일반 캐싱에서는 일관성 유지 작업이 단순| 각 노드가 동일한 최신 데이터를 공유할 수 있음.  <br/>데이터 동기화를 위한 프로토콜, 캐시 관리 전략 사용                  |

### 활용 사례
분산 캐싱을 사용하는 대표적 사례
- 웹 애플리케이션 : 사용자 세션, 데이터베이스 쿼리 결과 등 자주 접근하는 데이터를 저장하는데에 사용
- 데이터베이스 쿼리 결과 : ''
- 콘텐츠 전송 네트워크(CDN) : 엣지 서버를 활용하여 이미지, 동영상 등 정적 콘텐츠를 저장하고 제공. (CDN은 오리진 서버 데이터를 엣지 서버에 저장하두는 분산 캐싱 구조)
  - 오리진 서버 : 원본 데이터가 있는 서버
  - 엣지 서버 : 여러 지역에 배치된 캐시 서버
- 세션 관리 : 웹 애플리케이션의 세션을 효율적으로 관리하는 목적으로 활용 가능
- API 응답 캐싱 : 백엔드서버의 부하를 줄이고 응답 속도를 높이기 위해 API 응답 캐시 가능
- 실시간 분석 : 실시간 데이터 집계 등의 목적으로 활용 가능
- 메시지 큐 : 메시지 큐를 캐싱하고 중간 결과를 저장하여 메시지 처리 시스템의 효율성을 높이는 데에 활용 가능(메시지의 처리 과정이 여러단계일 때 중간 과정을 캐싱)


### 분산 캐싱을 사용할 때 장점
분산 캐싱을 사용하면 어떤 점이 좋나?
- 성능 개선 : 자주 사용하는 데이터를 메모리에 저장하여 데이터 접근 속도를 크게 높임
- 확장 용이성 : 부하가 증가하면 캐시 노드를 추가해서 쉽게 확장 가능
- 백엔드 시스템의 부하 감소 : 백엔드로 전달되는 요청 부하 감소 가능. DDos 공격 상황 등의 위험을 줄일 수 있음
- 장애 허용성 : 특정 노드에 장애가 발생해도 다른 노드가 데이터를 처리 가능하여 시스템 안정성 유지
- 일관된 접근 속도 : 자주 요청하는 데이터에 대해 일정한 속도로 접근 가능
- 비용 절감 : 백엔드 시스템의 부하 감소로 추가 서버, 리소스가 필요하지 않아 인프라 비용 절감 (=> 당연하지만 계산을 해봐야..)
- 사용자 경험 향상 : 응답 속도, 시스템 성능 개선으로 사용자 만족도 향상

### 분산 캐싱의 한계점
단점과 해결해야할 문제도 있다.
- 일관성 유지의 어려움 : 여러 캐시 노드에서 데이터를 일관되게 유지하는 것은 어려운 문제. 지연시간-일관성 간 타협이 필요할 수 있다
- 캐시 무효화의 복잡성 : 데이터가 변경될 때 캐시 데이터를 제거하거나 업데이트 하는 과정이 복잡할 수 있다. 모든 노드가 변경을 인지하고 갱신하게 보장하려면 추가적인 비용 발생 가능
- 설정과 관리의 복잡성 : 단일 노드 캐싱에 비해 설정 및 관리가 어려움. (노드 간 조율, 데이터 분산 전략 등)
- 네트워크 부하 문제 :  노드 간 업데이트를 주고받아야 하는 상황에서 네트워크 부하 초래 가능. (특히 지리적으로 분산된 경우)
- 캐시 데이터의 오래됨 문제 : 노드 간 동기화가 원활하지 않은 경우 업데이트 전 데이터를 반환하는 문제
- 데이터 분배의 어려움 : 데이터를 여러 노드에 적절히 나누려면 효과적인 분할 전략 필요. 잘못하면 일부 노드에 과부하, 일부 노드는 노는 문제
- 높은 메모리 사용량 : 캐시에 대량 데이터를 저장해야 하는 상황에서 메모리를 많이 사용해야하는 문제 발생 가능
- 비용 : 분산 캐싱에 필요한 인프라 및 유지 관리 비용 고려 필요. 분산 캐싱이 꼭 필요한지 비용 고려 필요
- 데이터 접근 패턴 : 캐시 히트율이 낮을때 캐싱 효과가 크지는 않은가?
- 쓰기 작업에서 한계 : 분산 캐싱은 읽기 작업이 많은 환경에 적합. 데이터를 자주 갱신하는 쓰기 작업에서는 노드간 데이터 일관성 유지 등에 어려움이 있을 수 있음.

결론은,  
분산 캐싱 도입 전 구체적인 요구사항을 살펴보고, 분산 캐싱이 정말 적절한 지 검토해봐야 한다.


## 6.2 분산 캐시 설계
분산 캐시를 설계해보자.  

요구사항을 정리하고, 구성 요소가 어떻게 동작하는지 살펴본다.

### 요구 사항 정의
요구사항을 기능적 요구사항, 비기능적 요구사항으로 나눌 수 있다.

#### 기능적 요구사항
- put(key, value) : 키-값 쌍을 캐시에 추가할 수 있다
- get(key) : 키로 값을 조회할 수 있다.

#### 비기능적 요구사항
- 높은 성능 : 지연시간은 짧아야 함. 처리량은 높아야 함. 시스템 전체의 응답속도를 개선하는 데 중요한 역할을 해야함.
- 높은 확장성 : 데이터와 요청, 사용자 수가 늘어나더라도 성능 저하 없이 대응할 수 있어야함. 필요한 경우 리소스나 노드를 추가해서 효율적으로 처리할 수 있아야 함
- 높은 가용성 : 장애가 발생하거나 노드 동작이 중단되더라도 서비스가 끊기지 않고 안정적으로 작동해야 함.

### 설계 과정
단계별로 캐싱 시스템을 만드는 과정을 거쳐보자.  

캐시에 적합한 데이터 구조를 선택하고,  
시스템 구성과 배포 전략을 구체화한다.

#### 캐싱에 필요한 데이터 구조
키와 값을 저장할 수 있는 가장 간단한 구조는 해시맵(또는 해시테이블)을 사용하는 것.  

만약 해시 맵 크기가 서버 메모리 용량을 초과한다면 어떻게 해야할까?  
=> 캐시 삭제 정책을 활용하여 문제 해결

#### 캐시 삭제 정책
`캐시 삭제`는 저장된 캐시 데이터 양이 설정된 용량에 도달하거나, 특정 조건이 충족되었을 때 어떤 데이터를 캐시에서 삭제할지 결정하는 과정을 의미.

어떤 정책들이 있는지 살펴보자.
- 삽입 순서 기반
  - 선입선출(FIFO) : 먼저 들어온 데이터 먼저 삭제하는 전략 (레스토랑 예시가 적합한지 의문)
  - 후입선출(LIFO)
- 접근 기반
  - 가장 최근에 사용한 항목 제거(Most Recently Used, MRU) : 최근에 사용한 항목이 당분간 다시 필요하지 않을 가능성이 높다고 판단되는 경우 효과적
  - 가장 오래된 항목 제거(Latest Recently Used, LRU) : 가장 일반적으로 사용하는 정책. 최근에 사용한 데이터를 다시 사용할 가능성이 높을 경우 효과적
  - 최소 사용 빈도 제거(Least Frequently Used, LFU) : 접근 빈도가 높은 데이터가 계속 접근 빈도가 높은 경우 유리. 약점은 과거에 자주 사용해서 빈도가 높으나 지금은 거의 사용되지 않는 경우 (ex, 음악 스트리밍 서비스에서 자주 재생한 곳을 캐시하는 등)

    
#### LRU 캐시 설계하기
많이 사용되는 LRU를 설계해보자

고려할점은
- 캐시에는 저장할 수 있는 항목의 수(N)가 정해져 있다.
- 항목을 추가하는 작업은 O(1)에 처리되어야 한다.
- 항목을 삭제하는 작업은 O(1)에 처리되어야 한다.

이중연결리스트와 해시맵을 조합한 데이터 구조를 살펴보자.

흐름을 살펴보자.

위 데이터 구조에서 `읽기`와 `쓰기` 두 가지 작업 흐름으로 구성된다.

쓰기 작업은 데이터베이스에 직접 저장한다고 가정한다.  
읽기 작업에서는 캐시에 데이터가 없을 때만 새로운 데이터를 추가한다.

읽기 작업의 흐름을 살펴본다.
주어진 키로 아래와 같은 형태의 valueObject를 찾아야 한다.
```json
valueObject : {
  "name" : "John Doe",
  "city" : "San Jose",
  "state" : "California"
}
```

다음 3가지 경우에서의 흐름을 살펴본다

#### 1. 키가 해시 맵에 없고 캐시의 항목 수가 N보다 적은 경우
캐시 미스 
새로운 항목을 추가해야 하며 캐시의 항목 수가 N보다 적은 상태.  
다음 단계를 따른다.
1. 데이터베이스에서 해당 키에 대한 valueObject를 가져온다
2. valueObject를 사용하여 노드를 생성하고 연결리스트 맨 앞에 추가한다.
3. 해시 맵에 키를 추가하고, 값에는 연결리스트의 해당 노드를 가리키는 포인터를 저장한다.

#### 2 키가 해시 맵에 없고 캐시가 이미 가득 찬 경우
캐시가 가득 찼기 때문에 공간을 확보해야한다
1. 연결리스트의 끝으로 가서 가장 오래 사용하지 않은 항목을 확인한다.
2. 연결 리스트에서 해당 항목을 삭제하고, 해시 맵에서도 이를 제거한다.
3. 1번 케이스와 동일한 단계를 수행한다.

#### 3. 키가 해시 맵에 있는 경우
1. 해시맵에서 키를 검색하여 해당 키의 포인터를 참고하여 해당하는 노드로 이동한다.
2. 해당 노드를 연결리스트의 맨 앞으로 이동시킨다.




### 시스템 통합하기
데이터 구조를 설계했다.  
이제 캐시를 효과적으로 배포하는 다양한 구성 방안을 살펴보자

#### 방법 1. 애플리케이션 서버와 캐시를 함께 배치하기
캐시를 애플리케이션 서버와 같은 머신에 함께 배치할 수 있다.  
애플리케이션 서버가 늘어날 때 캐시 인스턴스도 함께 추가되므로, 시스템 전체 처리 능력을 확장하는 데 적합하다.

- 장점 : 캐시 조회가 동일 머신 내 프로세스 간 호출로 하기 때문에 빠르다.  
- 한계 : 머신이 다운되면 요청은 다른 서버로 가지만 다운된 서버의 캐시 데이터는 없을 것이다.

#### 방법 2. 애플리케이션 서버와 독립적으로 구현하기
캐시를 애플리케이션 서버와 분리해서 독립적으로 운영되는 서버 클러스터에 배포하는 방식  
방법 1의 한계점을 해결하고 확정성을 고려한 설계를 가능하게 한다.

- 장점 : 요청 빈도가 증가하면 애플리케이션 서버와 상관없이 독립적인 확장이 가능하다. 
- 한계점 : 데이터를 나누어 저장해야 할 수도 있다. 이 때 키가 저장된 서버를 찾기 위해 로드밸런서, 키 검색 테이블 같은 방법을 고려해야한다.  
나머지 연산으로 분배할 수도 있고, 다른 해싱 전략을 사용하여 효율적이게 분배할 수도 있다.

#### 어떤 방법을 선택해야 할까?
애플리케이션의 요구 사항, 아키텍처, 우선순위를 종합적으로 고려해야 한다.

|             | 서버 통합형                                                  | standalone 형                                                |
|-------------|---------------------------------------------------------|-------------------------------------------------------------|
| 성능 요구 사항    | 애플리케이션 응답속도가 중요한 경우 네트워크 오버해드 없이 직접 접근 가능하여 좋을 수 있다.    | 확장성, 높은 처리량이 필요한 상황에서 유리할 수 있다.                             |
| 확장성         | 애플리케이션과 캐시를 함께 확장할 수 있는 상황에서 적합. 확장성 요구가 지나치게 높지 않을때 적합 | 캐시를 독립적으로 확장 가능하므로 더 높은 확장성을 제공                             |
| 자원 공유       | 애플리케이션 서버와 캐시가 자원을 공유하면 메모리, CPU를 더 효율적으로 활용 가능         | 캐시와 애플리케이션 서버의 자원 충돌이 발생하지 않아 대규모시스템에서 적합                   |
| 유연성         | 설정과 통합 과정이 간단하지만 기술 선택과 구성 옵션이 제한적                      | 요구사항에 따른 캐싱 솔루션 선택이 자유로움. 기술적으로 독립되어 유연함                    |
| 의존성 및 독립성   | 애플리케이션 서버와 연결되어 있어 서로 영향을 줄 수 있음                        | 독립적으로 동작하므로 시스템 전체에 미치는 영향이 상대적으로 적음                        |
| 운영 복잡성      | 배포와 관리가 비교적 간단하지만 캐시 사용에 있어 세부적인 요구사항 충족이 어려울 수 있음      | 설정과 관리가 비교적 복잡하나 애플리케이션 서버에 영향을 주지 않고 적잘한 방식으로 수정 및 업데이트 가능 |
| 인프라와 비용     | 애플리케이션 서버와 자원을 공유하기 때문에 인프라 비용이 적음                      | 캐시를 독립적으로 배치해야 하므로 추가 인프라 비용 발생                             |
| 네트워크 지연 허용성 | 네트워크 지연시간을 줄이는것이 최우선일때 적합                               | 캐시에 접근할 때 약간의 네트워크 지용을 허용할 수 있는 애플리케이션에 적합                  |


애플리케이션 요구 사항과 위 요소들을 종합적으로 고려해서 선택하는 것이 중요하다.

#### 요구 사항에 맞게 설계되었는지 점검하기
위에서 기능적 요구사항은 충족.  
비기능적 요구사항은?

- 높은 성능 : 모든 연산이 O(1)로 처리되어 효율적
- 높은 확장성 : 캐시를 배포하는 서버 수를 늘리고 이를 샤딩하면, 요청량 증가 및 동시성에 대응할 수 있어 확장성 측면에서 적합
- 높은 가용성 : 개선 필요. 한 서버가 다운되면 해당 서버에 저장된 데이터 전체를 잃을 위험

어떻게 높은 가용성을 챙길 수 있을까?  
=> 데이터 복제 활용

동일한 데이터를 여러 서버에 복제하고, 복제본을 다른 서버에 분산 저장하면 한 서버에 장애가 발생해도 다른 서버에서 요청을 처리할 수 있다.

데이터 복제를 활용하면 일관성 문제가 생길 수 있다.  
이 상황에서 가용성을 챙길지, 일관성을 챙길지 선택의 여지가 있다.

### 레디스와 맴캐시드 중 어떤 것을 선택해야 할까?
#### 사용목적
레디스는 다양한 데이터 구조릴 지원하여 활용 범위가 넓다.  
반면 맴캐시드는 단순 키-값 캐싱에서 속도가 빠르고 간단하게 사용할 수는 장점.

#### 데이터 보존 여부
레디스는 데이터를 영구적으로 저장할 수 있는 옵션 존재.  
맴캐시드는 메모리 기반 캐시라서 영구 저장 안됨

#### 다양한 데이터 구조 지원
레디스가 더 다양한 데이터 구조와 기능을 지원하므로 복잡한 데이터 조작이 필요한 경우 유리

#### 사용 편의성
맴캐시드는 단순하고 직관적인 설계 덕분에 쉽게 통합하고 운영할 수 있다는 장점  
레디스는 비교적 학습곡선이 높다

#### 기타
맴캐시드는 기본적으로 각 노드가 완전히 독립적이라 서로를 알지 못하며 데이터 복제가 불가능하다. 
레디스는 각 노드가 서로의 상태를 알고 있으며 복제와 마스터-슬레이브 구조로 마스터가 죽었을 때 failover를 지원한다.

맴캐시드는 데이터가 어떤 노드로 갈 지 클라이언트측에서 결정되는 반면,  
레디스는 서버사이드에서 결정된다.
